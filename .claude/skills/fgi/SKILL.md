# @fgi -- 가상 전문가 FGI (Focus Group Interview)

> **Trust Level: L1 (Draft)** - FGI 리포트 생성 후 PM 확인

## 이 스킬은 무엇인가요?

**Lenny's Podcast에 출연한 세계적 제품/성장 전문가들을 소환하여 PM의 기획 주제에 대해 가상 토론을 벌이는 도구**입니다.

전문가들이 서로 질문하고, 반론하고, 동의하며 **실제 패널 토론처럼 짧은 대화를 주고받습니다**. 모든 발언은 실제 트랜스크립트에 근거합니다.

> **쉽게 말해**: "크리에이터 대시보드 어떻게 만들지?" → Brian Chesky, Elena Verna, Casey Winters 등이 자기 경험을 바탕으로 토론해줍니다.

---

## 설계 원칙

- **Plan-first**: 전문가 발언 생성 전 트랜스크립트 섹션을 모두 Grep하여 근거를 확보합니다.
- **Tools-not-context**: 트랜스크립트를 SKILL.md에 인라인하지 않고, Grep/Read로 동적 조회합니다.
- **Bitter Lesson**: FGI 시뮬레이션 스크립트를 고정하지 않고, 주제에 따라 토론 방향을 모델이 판단합니다.
- **쉬운 말 원칙**: 신입 PM이 읽어도 바로 이해할 수 있는 대화체로 작성합니다.
- **CLI 가독성**: VSCode/터미널에서 읽기 좋은 출력. 소개는 blockquote 리스트, 대화는 인라인코드 헤더 + 볼드 핵심어, 출처는 라운드 끝 카운트, 결론은 쟁점 테이블 + 쉬운 해설. **HTML 태그 절대 금지** (`<details>`, `<summary>`, `<table>` 등) — 플러그인에서 렌더링되지 않는다. 마크다운만 사용한다.

## 워크플로우

### Step 1: 주제 분석 + 전문가 자동 선정 -- AI 전담 (PM 대기 없음)

#### 1a. 탐색 알고리즘 (4단계 순서 고정)

1. **키워드 추출**: PM 주제에서 검색 키워드 도출
   - 예: "크리에이터 관리 대시보드" → `growth-strategy`, `marketplaces`, `retention`, `product-management`
2. **토픽 파일 탐색**: `index/*.md` 파일명과 내용을 Grep하여 관련 토픽 파일 목록 확보
3. **게스트 목록 수집**: 관련 토픽 파일에서 게스트 이름 추출 (중복 제거, 여러 토픽에 등장하는 게스트 = 교집합 우선)
4. **frontmatter 선독**: 후보 게스트들의 트랜스크립트 frontmatter만 Read(limit=20)로 선독 → `guest`, `title`, `keywords`, `view_count` 확인

**다중 에피소드 처리**: 동일 전문가의 에피소드가 여러 개인 경우(예: `elena-verna`, `elena-verna-20` 등), 주제 키워드와 가장 높은 교집합을 가진 1개를 자동 선택하고, 선택한 에피소드를 소개에 기재한다.

#### 1b. 자동 선정 기준 (스코어링)

PM 선택 없이 AI가 자동으로 5~7명을 선정합니다:

| 기준 | 가중치 | 설명 |
|------|--------|------|
| 교집합 점수 | +2/토픽 | 여러 관련 토픽에 동시 등장하는 게스트 우선 |
| 키워드 매칭 | +1/키워드 | frontmatter keywords와 주제 키워드 교집합 |
| 조회수 보너스 | +1 (>50K) | 인기 에피소드 = 검증된 인사이트 |
| 관점 다양성 | 필수 | 동일 전문 분야 2명 이상 중복 방지 |

상위 5~7명 자동 선정 → **소개 카드** 출력 → 바로 Step 2 진행.
PM이 교체를 원하면 "교체" 발화로 요청 가능 (선택적).

#### 1c. 패널 소개 (blockquote 리스트 + "자세히" 후속 출력)

> **CLI 가독성 원칙**: 소개는 blockquote로 묶은 1줄 리스트로 빠르게 훑고, PM이 "자세히"라고 하면 펼침.

**기본 출력 — blockquote 리스트 (항상 출력):**

```
## 패널 ({N}명)

> **{이름}** — {한줄 전문성} · {소속} · {조회수}
```

**상세 출력 — PM이 "자세히" 발화 시 후속 메시지로 출력:**

각 전문가당 아래 요소를 **최대 5줄** 이내로 출력 (별도 메시지, HTML 태그 금지):

```
### April Dunford — 포지셔닝의 여왕

16개 제품의 포지셔닝을 재설계하여 매출 3배 성장시킨 B2B 전략가.
> *"고객이 왜 '지금 당장' 바꿔야 하는지를 말해야 합니다."*
**부른 이유**: 제품 특성 기반 경쟁 매핑 = 포지셔닝 문제.
**프레임워크**: Obviously Awesome — 경쟁 대안 → 차별화 속성 → 가치 → 타겟
```

**소개 작성 규칙:**
- **시그니처 인용**은 반드시 트랜스크립트 Grep으로 확보 (환각 금지)
- **크레덴셜**은 frontmatter의 description + 트랜스크립트 초반부에서 추출
- 펼침 카드는 **5줄 이내** 엄수 (크레덴셜 1줄 + 인용 1줄 + 부른이유 1줄 + 프레임워크 1줄)

PM이 "교체" / "바꿔줘" / "다른 전문가" 발화 시 → 대체 후보 제시 (선택적)

### Step 2: 전문가 관점 추출 -- AI 전담 (Agent 게스트당 1개 병렬)

각 Agent 내부 처리 순서 (**Grep-first 전략**):

1. frontmatter 재확인 (Read limit=20)
2. PM 주제 키워드로 트랜스크립트 본문 Grep (output_mode=content, -C=10)
3. 매칭된 섹션만 Read (offset/limit으로 청크, **전체 트랜스크립트 로드 금지** — 전체는 약 25K 토큰, 관련 섹션 2~5K 토큰이면 충분)
4. 추출물 생성:
   - 핵심 프레임워크/원칙 (최대 3개)
   - 관련 인용구 (원문 2~3개, 타임스탬프 포함)
   - 이 전문가의 입장/관점 요약 (1~2문장)
   - 예상되는 토론 입장 (이 주제에 대해 어떤 방향으로 발언할지)

전문가 브리프 카드 출력 → 바로 FGI 시작 (PM 대기 없음)

### Step 3: FGI 시뮬레이션 -- 하이브리드 (핵심)

> **토론 원칙**: 전문가들이 **서로에게** 질문하고 답변하며 토론합니다.
> 한 사람이 길게 말하지 않고, **짧은 대화를 주고받는** 실제 패널 토론 형식입니다.

#### 대화 방식 (다양한 유형 필수 포함)

| 유형 | 설명 | 예시 |
|------|------|------|
| **직접 질문** | 전문가 A가 전문가 B에게 질문 | "Casey, Eventbrite에서도 이런 문제 있었어요?" |
| **동의 + 보강** | 이전 발언에 동의하며 자기 경험 추가 | "맞아요, 저희도 Airbnb에서 비슷했는데..." |
| **정중한 반론** | 다른 관점 제시 | "저는 좀 다르게 봐요. 제 경험에서는..." |
| **구체화 요청** | 모호한 발언에 디테일 요청 | "그 '핵심 지표'가 구체적으로 뭐였어요?" |
| **프레임워크 제안** | 자기 프레임워크로 정리 | "제가 늘 쓰는 방법은 이거예요..." |
| **실패 경험 공유** | 실패 사례로 경고 | "저희는 그렇게 했다가 실패했어요..." |

#### Round 1 (AI 자동): 오프닝 토론 — 상호 질문/반론

PM의 주제에 대해 전문가들이 **서로에게** 질문하고 반론하며 토론합니다.

**대화 형식 — 인라인코드 헤더 + 볼드 핵심어:**

> **핵심 규칙**: 발언자/대상은 인라인코드(`` ` ``)로, 대화는 일반 텍스트 + **볼드** 핵심어.
> 첫 발언에는 `(전문성, 소속)` 풀 표기, 이후 재등장 시 `(전문성)`만 축약.

**예시:**

```
`April (포지셔닝 전문가, Obviously Awesome 저자)` → `전체`

이건 **포지셔닝 문제**예요. 속성 DB 만드는 건 좋은데,
"비슷한 상품 찾기"가 아니라 **"왜 이게 너의 경쟁인지"를 알려주는 도구**로 써야 해요.

`Dan (마켓플레이스 성장, Thumbtack Growth)` → `April (포지셔닝)`

동의요. 근데 **500개 한번에 매칭하면 안 돼요.**
세럼이면 세럼끼리, **카테고리 안에서만** 시작하세요.

`Elena (AI 성장 전략, Lovable $200M ARR)` → `Dan (마켓플레이스)`

순서가 틀렸어요. 이 리포트가 **브랜드를 데려오는 미끼**예요.
**먼저 줘야 거래가 시작**돼요. 기다리면 아무도 안 와요.*

출처 3건 · 추론 1건(*)
```

**출력 포맷 규칙:**
- 발언자: `` `이름 (전문성, 소속)` → `대상 (전문성)` `` — 인라인코드로 감싸기
- 대상도 인라인코드 + 괄호 전문성 표기 (`` `전체` ``는 괄호 없음)
- 첫 등장 시 `(전문성, 소속)` 풀 표기. 재등장 시 `(전문성)`만 축약
- 대화 본문: 일반 텍스트 + **볼드**로 핵심 인사이트 강조
- `*` 마크 — 추론 발언 (트랜스크립트 근거 없음)
- 라운드 끝에 `출처 N건 · 추론 N건(*)` 카운트
- **발언 사이 빈 줄 필수**: 각 전문가 발언 블록 사이에 반드시 빈 줄(blank line)을 삽입하여 시각적으로 구분한다
- 한 발언 = **최대 3줄**. 쉽고 간결한 문장. 전문 용어 최소화

#### PM 추가 질문 (멀티턴)

PM이 모더레이터로서 추가 질문 → 전문가들이 같은 대본 스타일로 응답.
PM이 원하는 만큼 질문 계속.

PM이 "정리해줘" / "종료" / "wrap up" 발화 시 → Step 4로 이동

## 대화 생성 규칙

### 1. Grounding Rule (라운드 끝 카운트)

> **인라인 출처 금지**. 출처는 라운드 끝에 카운트로 표기합니다.

**발언 중 표기:**
- 트랜스크립트 근거 있음 → 아무 표기 없음
- 트랜스크립트 근거 없음 → 발언 끝에 `*` 마크

**라운드 끝 카운트:**
```
출처 N건 · 추론 N건(*)
```

- 상세 출처는 결론 리포트의 출처 테이블에 모아서 표기
- `*` 추론 발언은 대화 본문에서 `*` 마크로 식별 가능

### 2. 대화체 규칙

- **짧게**: 한 발언 **최대 3줄**. 긴 연설 금지.
- **발언 사이 빈 줄**: 각 전문가 발언 블록 사이에 반드시 **빈 줄(blank line)**을 삽입한다. 발언이 붙어 있으면 누가 말하는지 구분이 안 된다.
- **쉽게**: 신입 PM이 사전 없이 이해할 수 있는 문장. 전문 용어 최소화.
- **볼드 핵심어**: 각 발언에서 가장 중요한 인사이트를 **볼드**로 강조.
- **상호작용**: `` `이름 (전문성)` → `대상 (전문성)` `` 형식으로 누가 누구에게 반응하는지 명시.
- **다양한 유형**: 한 라운드에 질문/동의/반론/구체화요청/경험공유 중 최소 3가지 유형 포함.
- **이름만**: 성 생략, First name만 사용.

### 3. 추론 비율 상한

한 라운드에서 `[추론]` 태그 비율이 50%를 초과하면:
```
[알림] 이 라운드에서 추론 비율이 높습니다 ({N}%).
트랜스크립트 근거가 부족한 주제일 수 있습니다.
다른 각도로 질문하시거나, 전문가를 교체하시겠습니까?
```

### 4. 쟁점 보존

전문가 간 의견 충돌을 AI가 임의로 해소하지 않습니다.
쟁점은 쟁점으로 남겨두고 PM 판단에 맡깁니다.

파이프라인 외부의 독립 도구다. PM이 수시로 호출하고, 결과 인사이트는 prd-architect·scenario-designer·jtbd-designer 등에 PM이 수동으로 참고 반영한다.

## 금지된 작업

1. 트랜스크립트 근거 없는 전문가 발언 생성 (출처 태그 누락 시 자동 `[추론]` 분류)
2. 전문가의 실제 입장과 반대되는 발언 생성
3. 소개 카드 없이 바로 토론 진행 (PM이 누군지 모르는 상태에서 토론 시작 금지)
4. 5명 미만 패널로 FGI 진행
5. 전문가 간 의견 충돌을 AI가 임의로 해소
6. 트랜스크립트 전체를 통째로 Read (**Grep-first 필수**)
7. 한 전문가가 3문장 이상 연속 독백 (짧은 대화 원칙)
8. 존재하지 않는 게스트 슬러그 사용 시 무시하고 진행 (에러 안내 필수)
9. **HTML 태그 사용 금지** — `<details>`, `<summary>`, `<table>`, `<div>` 등 모든 HTML 태그 사용 금지. 플러그인/CLI에서 렌더링되지 않는다. 마크다운(##, >, |, **, `` ` `` 등)만 사용한다
