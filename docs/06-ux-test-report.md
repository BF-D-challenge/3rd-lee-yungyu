# 오늘 해볼까 첫 화면 합성 UX 테스트

- 일자: 2026-07-16
- 범위: 아이디어 만들기 첫 화면 1개
- 방법: `$ux-tester` 기본 페르소나 3명 독립 리뷰 + 구조 리뷰 1회
- 증거 수준: L1 합성 가설. 실제 사용자 관찰 결과가 아니며 출시 판단을 대체하지 않는다.
- 입력: 현재 구현을 정규화한 ASCII 화면. 인앱 브라우저 제어가 로컬 URL 정책에 막혀 이번 회차에는 실제 화면 녹화 대신 코드·E2E와 ASCII를 사용했다.

## 테스트 과업

1. 도움 없이 첫 카드를 뽑는다.
2. 자동 채우기와 카드 사이 3초 대기의 의미를 이해한다.
3. 네 장 완성 후 결과 화면으로 이동하고 아이디어를 설명한다.

## 1차 결과

| 페르소나 | 추정 시간 | 과업 완료 | 관찰 가설 |
|---|---:|---:|---|
| 신입 사용자 | 00:55 | 2/3 | `여기에 카드 놓기`를 드래그 전용 안내로 해석해 첫 탭을 시행착오로 발견했다. |
| 중급 사용자 | 00:51 | 1/3 | 빈 슬롯과 덱 중 어디를 먼저 조작해야 하는지 확신하기 어려웠다. |
| 숙련 사용자 | 00:29 | 2/3 | 탭 경로를 찾았지만 가장 빠른 행동이 문구만으로 명확하지 않았다. |

시간과 완료율은 합성 리뷰의 상대 비교용 가설이다. 결과 화면 시각 자료가 입력에 포함되지 않아 3번 과업의 이해도는 판정하지 않았다.

## 합의된 문제

- 3/3 페르소나와 독립 구조 리뷰가 첫 행동의 탭 가능성이 약하다고 판단했다.
- 2/3 페르소나는 자동 채우기와 3초 대기 중 현재 상태를 더 예측 가능하게 보여줄 여지가 있다고 판단했다.
- 숙련 사용자 1/3은 3초를 느리게 볼 수 있다고 예상했지만, 읽을 시간을 보장한다는 제품 결정에 따라 3초는 유지했다.

## 적용한 개선

- 빈 슬롯의 대표 문구를 `눌러서 카드 뽑기`로 변경했다.
- 보조 문구로 `아래 카드를 끌어 놓아도 돼요`를 두어 기존 드래그 인터랙션을 유지했다.
- 접근 가능한 이름도 `{축 이름} 카드 뽑기`로 맞췄다.
- 카드 뽑기 전후 가이드 패널 높이를 216px로 고정해 상태 전환 때 레이아웃이 흔들리지 않게 했다.

## 폐쇄 루프 재평가

동일한 세 페르소나에게 수정된 첫 화면을 다시 독립 평가시켰다.

| 페르소나 | 추정 시간 | 첫 카드 과업 | 남은 가설 |
|---|---:|---:|---|
| 신입 사용자 | 약 6초 | 성공 | `검증된 원본` 용어와 복수 진입점이 잠시 헷갈릴 수 있다. |
| 중급 사용자 | 약 4초 | 성공 | 카드 뽑기 후 되돌리기 가능 여부를 확인하고 싶을 수 있다. |
| 숙련 사용자 | 약 2~3초 | 성공 | 탭·드래그 안내를 함께 보면 아주 짧은 선택 망설임이 생길 수 있다. |

판정: 첫 행동 모호성은 3/3 시행착오 가설에서 3/3 직접 성공 가설로 개선됐다. 남은 항목은 실제 사용성 테스트에서 관찰한 뒤 추가 변경한다.

## 실제 사용자 검증안

- 참여자: 제품을 처음 보는 5명
- 성공 기준:
  - 4/5 이상이 도움 없이 10초 안에 첫 카드를 뽑는다.
  - 4/5 이상이 자동 채우기와 카드 사이 3초 대기의 목적을 설명한다.
  - 4/5 이상이 4/4 이후 결과를 찾고 한 문장으로 설명한다.
- 관찰 질문:
  1. 처음 어디를 눌렀나요?
  2. 자동 채우기 전에 무엇이 걱정됐나요?
  3. 4/4 이후 무엇을 확인하거나 바꾸고 싶었나요?

## 현재 판정

첫 화면의 대표 행동은 구현·합성 재평가 기준으로 개선됐다. 자동 진행의 예측 가능성과 결과 화면 이해도는 실제 사용자 5명 검증 전까지 미검증 가설로 남긴다.
