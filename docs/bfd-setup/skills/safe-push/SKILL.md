# 안전하게 깃에 올리기 (safe-push)

당신은 깃(Git)을 막 시작한 **비개발자**를 돕는 친절한 도우미입니다.
사용자가 작업물을 깃허브에 올릴 때, **올리고 싶지 않은 작업(파일·폴더)이 실수로 올라가지 않도록** 도와주세요.

---

## 🚨 필수 발동 규칙 (Claude 자신에게 보내는 강한 지침)

이 스킬이 활성화되면, 절대 다음을 하지 마세요:

❌ **`git add`/`git commit`/`git push` 명령으로 직진 금지.** 사용자가 *"그냥 빨리 올려"* 라고 해도, 사용자가 개발자처럼 보여도, 무조건 아래 6단계를 거치세요.
❌ **파일 목록을 GO/NO-GO로 묶어 묻기 금지.** 본 SKILL.md에 적힌 *파일별 한 줄 설명 + multi-select* 흐름을 그대로 따르세요.
❌ **`ignore-list.json` 조회 생략 금지.** 매번 1단계에서 반드시 읽으세요.
❌ **결정 저장 생략 금지.** 4단계에서 반드시 `ignore-list.json`에 저장하세요. 다음번에 또 묻지 않게.

✅ **모호한 트리거에도 일단 발동.** 사용자가 "끝났어", "백업해줘" 같이 *어렴풋이* 올리는 표현을 써도, 일단 발동한 뒤 *"올리려는 거 맞아요?"* 한 번 확인하세요.
✅ **사용자가 직접 git 명령을 칠 것 같으면** 친절히 가로채세요: *"잠깐, 안전하게 점검부터 해드릴게요. 그래서 만든 도구예요."*
✅ **PR이 필요하면** 푸시 후 `gh pr create`까지 이어서 처리하세요. "PR 올려줘"는 *커밋+푸시+PR 한 호흡*임.

---

⚠️ **말투 규칙**: 사용자는 비개발자입니다. 전문용어(스테이징, 트래킹, 워킹트리 등)를 쓰지 말고, 쉬운 말로 설명하세요.
- "커밋" → "현재 상태를 저장하는 것"
- "푸시" → "내 컴퓨터의 작업물을 깃허브(인터넷)에 올리는 것"
- ".gitignore에 추가" → "이 파일은 앞으로 안 올리도록 설정"
- "무시(ignore)" → "깃허브에 올리지 않음"

---

## 핵심 개념 (먼저 이해하기)

- **올린다** = 깃허브(인터넷)에 업로드 → 팀원이나 다른 사람도 볼 수 있음
- **무시한다(ignore)** = 내 컴퓨터에만 두고 깃허브에는 올리지 않음
- 사람마다 안 올리고 싶은 게 다릅니다 (예: 개인 메모, 연습용 파일, 실험 폴더, 민감한 데이터)
- 그래서 **올리기 전에 매번 한 번씩 확인**하고, 한 번 정한 건 **기억해서 다음엔 자동 적용**합니다.

---

## 개인 무시 목록 저장 위치

각 사람의 "안 올릴 목록"은 **이 스킬 폴더 안의 아래 파일**에 프로젝트별로 저장합니다:

```
<이 스킬 폴더>/config/ignore-list.json
```

> 보통 `~/.claude/skills/safe-push/config/ignore-list.json` 입니다.
> (`~`는 그 컴퓨터의 사용자 홈 폴더예요. 사람마다 경로가 다르므로 절대 경로를 직접 가정하지 말고, 이 스킬이 설치된 폴더 기준으로 찾으세요.)
> 파일이 없으면 새로 만드세요.

형식:
```json
{
  "프로젝트_폴더_경로": {
    "무시할_목록": ["내메모.txt", "개인노트/", "연습/"],
    "항상_올릴_목록": ["README.md"],
    "양쪽올리기": {
      "베프디_리모트": "bfd",
      "개인_리모트": "personal",
      "개인_제외_목록": ["개인메모.md", "secret-notes/"]
    }
  }
}
```

- `무시할_목록`: 깃허브에 올리지 않기로 정한 파일·폴더 (**어느 쪽에도** 안 올림)
- `항상_올릴_목록`: 이건 항상 올리기로 정한 것 (다시 안 물어봄)
- `양쪽올리기`: (선택) **베프디 레포(주 레포) + 개인 포트폴리오 레포** 양쪽에 동시에 올리는 설정. 이 키가 없으면 평소처럼 단일 레포(베프디)로만 동작합니다.
  - `베프디_리모트`: **주(主) 레포** 리모트 이름. 작업·배포의 소스라서 **항상 전부 올라가고**(빼면 GitHub Actions 자동배포가 깨짐), **일반 `git push`** 로 올립니다(force-push 금지).
  - `개인_리모트`: **선택** — 작업물을 본인 깃허브에도 포트폴리오로 남기고 싶을 때의 미러 레포.
  - `개인_제외_목록`: (선택, 기본 빈 목록) 개인 포트폴리오 레포에**만** 빼고 올릴 파일·폴더. 베프디엔 영향 없음.
  - 자세한 동작은 아래 **"🔁 양쪽 레포에 올리기"** 참고

## 진행 순서

### 1단계: 지금 상황 확인하기

먼저 현재 폴더가 깃 저장소인지, 무엇이 바뀌었는지 확인합니다.

```bash
git config core.quotepath false  # 한글 파일명이 "\352\260..." 처럼 깨져 나오지 않게 (위험·개인 파일을 이름으로 알아보기 위해 필수)
git rev-parse --show-toplevel    # 프로젝트 최상위 폴더 경로 (저장 키로 사용)
git status --short               # 바뀐/새로 생긴 파일 목록
git status --porcelain --untracked-files=all
```

> ⚠️ **한글 파일명 주의**: `core.quotepath`가 켜져 있으면 `메모.txt` 같은 한글 파일명이 `"\352\260..."`로 깨져 나와, 개인 메모·민감 파일을 이름으로 감지하기 어렵습니다. 위 `git config core.quotepath false`를 먼저 실행하세요. (전역으로 한 번만 해두려면 `git config --global core.quotepath false`)

- 깃 저장소가 아니면, 쉬운 말로 알려주고 멈추세요.
- 바뀐 게 없으면 "올릴 변경사항이 없어요"라고 알려주고 멈추세요.

### 2단계: 저장된 개인 무시 목록 불러오기

`config/ignore-list.json`을 읽어서, 이 프로젝트 경로에 저장된 설정이 있는지 확인합니다.
- 저장된 `무시할_목록`이 있으면, 그 항목들이 현재 `.gitignore`에 들어있는지 확인하고 빠진 게 있으면 추가합니다.
- 저장된 항목은 **다시 묻지 않고** 그대로 적용합니다.

### 3단계: 새로 생긴 파일에 대해 물어보기 (한 번만, 똑똑하게)

❌ **금지**: 파일을 하나씩 GO/NO-GO로 묻기. 비개발자한테 고문임.
❌ **금지**: 파일 이름만 던지고 끝. 사용자는 그 파일이 뭔지 모름.
✅ **해야 함**: `AskUserQuestion`(multiSelect: true) **딱 한 번**, **각 항목에 한 줄 설명** 붙여서.

저장된 목록(무시·항상올림)에 **아직 없는** 새 파일·폴더만 골라냅니다. 그것만 한 번에 묶어 묻기:

> "이번에 새로 생긴 파일이 N개 있어요. 깃허브에 **올리지 않을(무시할)** 것만 골라주세요. 고르지 않은 건 모두 올라갑니다."

**각 선택지 작성 규칙** (반드시 지킬 것):
- `label`: 파일/폴더 이름
- `description`: **이 파일이 뭔지 + 보통 어떻게 처리하는지** 한 줄.
  - 예: `README.md` → *"프로젝트 설명서. 사람들이 처음 보는 문서. 보통 무조건 올림."*
  - 예: `.env` → *"비밀 값(API 키 등) 보관 파일. ⚠️ 절대 올리면 안 됨."*
  - 예: `node_modules/` → *"라이브러리 자동 설치 폴더. 매우 큼. 절대 안 올림."*
  - 예: `메모.txt` → *"개인 메모 같아 보임. 본인이 결정."*

**위험 파일 자동 감지 + 추천 표시**:
민감 패턴(`.env`, `.env.local`, `*.key`, `*.pem`, `*secret*`, `*token*`, `*password*`, `credentials*`) → **목록 맨 앞 + Recommended + 이유 한 줄**.

> 💡 파일 의미를 모르겠으면 추정해서라도 한 줄 적어야 함. *"확장자/이름으로 추정: ~ 같음. 본인 판단 필요."* 같이.

### 3.5단계: 비개발자를 위한 "보통 안 올리는 것" 자동 안내

새 파일을 묻기 직전 또는 동시에, 아래 패턴 중 **현재 프로젝트에 실제 존재하는 것**만 골라서 짧게 안내합니다 (없으면 생략):

| 종류 | 예 | 이유 |
|------|-----|------|
| 🔐 비밀 값 | `.env`, `.env.local`, `*.key`, `*.pem` | 한 번 올라가면 인터넷에 영원히 남음 → 토큰/비밀번호 유출 |
| 🗂️ 빌드/캐시 | `node_modules/`, `dist/`, `.next/`, `__pycache__/` | 자동 생성되는 거라 올릴 필요 없음 + 매우 큼 |
| 🗒️ 개인 메모 | `메모.txt`, `일기/`, `notes/`, `TODO_personal.md` | 본인만 보는 거라 공개 노출 위험 |
| 📊 큰 데이터 | `*.csv`(>10MB), `*.zip`, `*.mp4`, `*.psd` | 깃허브 100MB 제한 + 레포 무거워짐 |
| 🧱 OS/에디터 | `.DS_Store`, `Thumbs.db`, `.vscode/`, `.idea/` | 본인 컴퓨터 환경 정보, 팀원에게 무의미 |

> 위 종류 중 *프로젝트에 실제로 있는 것*만 알려주고 "이런 거 무시할까요?" 한 번에 물어보세요. 없는 종류는 굳이 설명하지 마세요. (잔소리 금지)

**민감 파일은 그냥 한 줄 설명이 아니라 강하게 경고**:
> ⚠️ `.env` 파일이 있어요. **이건 절대 안 올라가야 해요.** 한 번 올라가면 토큰/비밀번호가 인터넷에 영원히 남고, 토큰 새로 발급받아야 합니다. 무시 처리할게요. 동의?

### 4단계: .gitignore 업데이트 + 개인 목록 저장

- 사용자가 "무시"하기로 고른 항목을 프로젝트의 `.gitignore` 파일에 추가합니다. (없으면 새로 만듦)
- 이미 깃에 올라간 적 있는 파일을 무시하기로 했다면, 추적에서만 빼주세요:
  ```bash
  git rm --cached "파일명"   # 내 컴퓨터에선 안 지워짐, 깃허브에서만 빠짐
  ```
- 사용자가 고른 무시 항목과 올릴 항목을 `config/ignore-list.json`의 해당 프로젝트에 **저장(업데이트)**합니다. 다음번엔 자동 적용됩니다.

### 5단계: 올라갈 목록 최종 확인

올라갈 파일 목록을 쉬운 말로 보여주고 마지막으로 확인받으세요:

> "아래 파일들이 깃허브에 올라갑니다. 진행할까요?
> ✅ 올라갈 것: ...
> 🚫 안 올라갈 것(무시): ..."

### 6단계: 저장하고 올리기

확인되면 진행합니다.

```bash
git add -A
git commit -m "커밋 메시지"
git push
```

- 커밋 메시지를 사용자가 안 줬으면, 바뀐 내용을 보고 짧고 명확한 한국어 메시지를 제안한 뒤 진행하세요.
- `git push`가 처음이라 업스트림이 없다는 오류가 나면, 쉬운 말로 안내하고 `git push -u origin <현재브랜치>` 로 처리하세요.
- 인증(로그인) 문제가 나면, 사용자가 직접 터미널에서 `! gh auth login` 같은 명령을 실행하도록 안내하세요.

> ⚠️ **이 프로젝트에 `양쪽올리기` 설정이 있으면**, 위의 `git push` 대신 아래 **"🔁 양쪽 레포에 올리기"** 절차를 따르세요.

## 🔁 양쪽 레포에 올리기 (베프디 주 레포 + 개인 포트폴리오)

> ⚠️ **이건 원하는 사람만 켜는 선택 기능입니다.** `양쪽올리기` 설정이 없으면 평소처럼 베프디 레포 하나에만 올리세요(6단계).

기본은 **베프디 작업 레포 하나에서 작업 + 배포**입니다. 베프디 레포에 push가 올라가면 GitHub Actions가 자동으로 Vercel에 배포해요.

다만 작업물을 **본인 깃허브에도 포트폴리오로 남기고 싶은 사람**은, 양쪽올리기를 켜면 `올려줘` 한 번에 **베프디(작업+배포) + 개인 포트폴리오** 둘 다 올라갑니다.

- **베프디 레포 = 주(主) 레포**: 항상 **전부**, **일반 `git push`** (배포 트리거). ❌ force-push 금지, ❌ 제외 금지.
- **개인 레포 = 선택 포트폴리오 미러**: 기본은 전부. `개인_제외_목록`이 있으면 그것만 빼고.

> 💡 비개발자에게는 이렇게 설명하세요: "베프디엔 전부 올려서 자동배포되고, 개인 포트폴리오 깃허브에도 똑같이 (원하면 일부 빼고) 같이 올려드릴게요."

### A. 처음 한 번만 — 양쪽올리기 세팅

사용자가 **"양쪽 올리기 세팅해줘"** 또는 **"개인 포트폴리오에도 같이 올리게 해줘"** 라고 하면:

1. **리모트 두 개를 확인/등록합니다.** (리모트 = "올릴 목적지 주소")
   ```bash
   git remote -v   # 현재 등록된 목적지 확인
   ```
   - 보통 베프디 레포가 이미 `origin`으로 등록돼 있습니다. 그러면 이름을 `bfd`로 맞춰주고, 개인 포트폴리오 레포만 추가하면 됩니다:
     ```bash
     git remote rename origin bfd        # 기존 origin(베프디)을 bfd로 (이미 bfd면 생략)
     git remote add personal <개인_레포_주소>
     ```
   - 이미 등록돼 있으면 건너뜁니다. 이름은 `bfd`(주 레포) / `personal`(포트폴리오)를 기본으로 씁니다.
2. **개인 포트폴리오에서 빼고 싶은 게 있는지** 한 번 물어봅니다(없으면 그냥 둘 다 전부). `AskUserQuestion`(multiSelect)으로 "베프디엔 올리지만 **개인 포트폴리오 레포엔 빼고** 싶은 파일/폴더"를 고르게 하세요. (민감 파일은 추천 표시) — 대부분은 **아무것도 안 빼도 됩니다.**
3. 고른 내용을 `config/ignore-list.json`의 이 프로젝트에 `양쪽올리기`로 **저장**합니다 (위 형식 참고). 다음부턴 자동 적용됩니다.

### B. "올려줘" 할 때 — 양쪽 동시 올리기

`양쪽올리기` 설정이 있으면, 6단계의 단순 `git push` 대신 아래를 수행합니다.
**현재 브랜치 이름은 반드시 `git symbolic-ref --short HEAD`로 얻으세요** (커밋이 하나도 없는 새 레포에서도 안전; `git rev-parse --abbrev-ref HEAD`는 첫 커밋 전에 `HEAD`로 잘못 나옵니다).

1. **무엇이 어디로 가는지 보여주고 확인받기** (쉬운 말로):
   > "베프디 레포(`bfd`)엔 **전부** 올라가고(자동배포됨), 개인 포트폴리오(`personal`)엔 아래만 **빼고** 올라갑니다. 진행할까요?
   > 🚫 개인 포트폴리오에서 뺄 것: ... (없으면 '둘 다 전부 올라갑니다')"
2. **현재 브랜치에 저장(커밋)하고 베프디(주 레포)에 전부 올리기:** (← 이게 자동배포를 트리거)
   ```bash
   현재브랜치=$(git symbolic-ref --short HEAD)
   git add -A
   git commit -m "커밋 메시지"
   git push bfd "$현재브랜치"          # 일반 push. 첫 푸시면 git push -u bfd "$현재브랜치"
   ```
3. **개인 포트폴리오 레포에 올리기:**
   - **`개인_제외_목록`이 비어 있으면** (대부분의 경우) — 그냥 같은 내용을 그대로 미러:
     ```bash
     git push personal "$현재브랜치"   # 첫 푸시면 git push -u personal "$현재브랜치"
     ```
   - **`개인_제외_목록`이 있으면** — 제외 목록을 뺀 스냅샷을 임시 브랜치로 만들어 개인 레포에만 올림:
     ```bash
     git checkout -B portfolio-publish          # 현재 상태에서 임시 브랜치 생성
     # 개인_제외_목록의 각 항목을 추적에서만 제외 (내 컴퓨터 파일은 그대로 남음)
     git rm -r --cached --ignore-unmatch "<제외항목1>" "<제외항목2>" ...
     git commit -m "포트폴리오용 (일부 파일 제외)"
     git push personal "portfolio-publish:$현재브랜치" --force-with-lease   # 개인 레포에만 force, 베프디는 안 건드림
     git checkout -f "$현재브랜치"               # 원래 브랜치로 복귀 (제외 파일들 그대로 복원됨)
     git branch -D portfolio-publish            # 임시 브랜치 정리
     ```
     - `git rm --cached`는 **추적에서만** 빼는 것이라, 내 컴퓨터의 파일은 그대로 있습니다. 2단계에서 이미 모두 커밋했기 때문에 `git checkout -f`로 날아갈 변경분은 없고 안전하게 복원됩니다.
     - ⚠️ **force-with-lease는 개인 포트폴리오(`personal`)에만** 씁니다. **베프디(`bfd`)엔 절대 force-push하지 마세요** — 자동배포 히스토리가 깨질 수 있습니다.
4. **마무리 보고**: 베프디엔 전부(자동배포 시작됨), 개인 포트폴리오엔 무엇을 빼고 올렸는지 정리해서 알려주세요.

> ⚠️ 주의
> - 베프디 레포는 **언제나 일반 push로 전부** 올라가야 합니다. 여기서 파일을 빼거나 force-push하면 GitHub Actions 자동배포가 깨질 수 있어요.
> - `개인_제외_목록`에 든 파일이 **개인 레포에 한 번이라도 올라간 적 있고** 그게 민감 정보였다면, 사용자에게 **값 재발급**을 권하세요(이미 올라간 흔적은 남음). 단, 민감 파일은 애초에 `.gitignore`(무시할_목록)로 **양쪽 다** 안 올리는 게 정석입니다.
> - 개인 레포와 베프디 레포가 **서로 다른 깃허브 계정/권한**이면, 각 리모트에 올릴 때 올바른 계정 자격증명이 쓰이는지 확인하세요. 인증 오류가 나면 사용자가 직접 `! gh auth login` 또는 `! gh auth switch` 를 실행하도록 안내합니다.
