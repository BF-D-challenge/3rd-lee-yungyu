# 공통 실제 검증 프로토콜

## 1. 같은 조건으로 비교한다

각 후보는 아래 순서만 사용한다.

```text
모집 메시지 노출
→ 한 화면 랜딩
→ 20분 인터뷰
→ 실제 자료 1건으로 수기 결과 제공
→ 가격 제시
→ 결제/예약금 또는 거절 이유 기록
```

랜딩만 보고 결제한 사람과 인터뷰 뒤 결제한 사람은 구분해 기록한다.

## 2. 최소 계측

| 단계 | 이벤트 | 필수 속성 |
|---|---|---|
| 유입 | `landing_view` | 후보 ID, 유입 채널, 시각 |
| 관심 | `primary_cta_click` | 후보 ID, CTA 문구, 가격 |
| 인터뷰 | `interview_booked` | 후보 ID, 채널, 적합 여부 |
| 실제 문제 | `recent_problem_confirmed` | 마지막 발생일, 빈도, 손실 |
| 수기 결과 | `manual_result_delivered` | 소요시간, 입력 수, 결과 사용 여부 |
| 결제 | `payment_started` | 가격, 결제 방식 |
| 결제 완료 | `payment_completed` | 가격, 환불 가능 여부 |
| 거절 | `payment_declined` | 고정 거절 코드, 자유 메모 |
| 재사용 | `repeat_use` | 최초 사용 후 경과일 |

## 3. 결제 거절 코드

- `NO_PAIN` — 최근 실제 문제가 없음
- `LOW_FREQUENCY` — 너무 가끔 발생
- `FREE_ALTERNATIVE` — 무료 도구로 충분
- `PLATFORM_SOLVED` — 기존 플랫폼 기본 기능으로 충분
- `GENERIC_AI` — ChatGPT·Claude로 충분
- `PRICE` — 필요하지만 가격이 높음
- `TRUST_PRIVACY` — 자료 제공·개인정보가 불안
- `WORKFLOW_FRICTION` — 입력·공유 과정이 번거로움
- `NOT_DECISION_MAKER` — 본인이 결제자가 아님
- `OTHER`

## 4. 가격 제시 규칙

- 인터뷰 중간에 가격을 흘리지 않는다.
- 현재 방식과 최근 손실을 먼저 확인한다.
- 수기 결과를 실제로 전달한 뒤 정해진 가격 하나만 제시한다.
- 할인과 가격 흥정으로 통과 수를 만들지 않는다.
- 결제 취소·환불 경로를 결제 전에 명확히 고지한다.

## 5. Go / Hold / Kill

### Go

- 후보별 결제 기준 충족
- 최근 실제 문제 인정 기준 충족
- 플랫폼 기본 기능·범용 AI가 핵심 대안이 아님
- 수기 제공 시간이 1인 운영 가능한 수준

### Hold

- 문제 인정은 강하지만 결제 기준보다 한 명 부족
- 가격 또는 타겟 한 가지만 수정하면 재검증 가능
- 최대 한 번만 수정 실험

### Kill

- 실제 결제 0명
- 최근 문제 경험자가 절반 미만
- 과반이 플랫폼 기본 기능 또는 범용 AI로 충분하다고 답함
- 개인정보·법률·금전 위험을 제거하면 핵심 가치도 사라짐

## 6. 비교 공정성

- 후보마다 모집 채널을 최소 2개 사용한다.
- 유효 인터뷰 수를 채우기 전 단순 클릭률로 순위를 정하지 않는다.
- 서로 다른 기간의 광고비·노출 수를 그대로 비교하지 않는다.
- 최종 비교는 `유효 인터뷰당 결제자 비율`과 `수기 결과 1건 제공 시간`을 함께 본다.
