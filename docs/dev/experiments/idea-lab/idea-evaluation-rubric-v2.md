# 아이디어 공통 평가표 v2.1 — B2C + 1인 사업자 B2B

적용일: 2026-07-12  
목적: 아이디어의 사용성, 시장성, 사용 순간 접근성, 포트폴리오 중복을 서로 다른 판단으로 분리한다.

## 1. 세 개의 판단을 섞지 않는다

| 판단 | 질문 | 담당 기준 | 결과 |
|---|---|---|---|
| UX | 처음 본 사람이 이해하고 시작할 수 있는가? | 고정 P1/P2/P3 | UX Pass/Fail |
| 제품성 | 누가 왜 돈을 내고, 혼자 만들 수 있는가? | 수익화·GTM 점수 | Product Pass/Hold/Fail |
| 사용 순간 접근성 | 제휴 없이 필요한 순간의 사용자를 만날 수 있는가? | 사전 기록 필요성·첫 20명 도달 경로 | Moment 2/1/0 |
| 포트폴리오 | 기존 그룹과 같은가, 다른가? | 입력·타겟·순간·결과 비교 | New/Merge |

`결제할 것 같다`는 합성 UX의 과업이 아니다. 합성 사용자는 화면 이해와 행동 가능성을 평가하고, 결제는 별도 시장 가설로 평가한다.

## 2. UX 하드 게이트

같은 ASCII 카드에서 고정 경험 3층이 다음 과업을 독립 평가한다.

1. 10초 안에 누구의 어떤 문제인지 설명할 수 있다.
2. 첫 입력 또는 첫 행동이 무엇인지 찾을 수 있다.
3. 완료 뒤 받는 결과를 설명할 수 있다.

두 명 이상이 세 과업을 모두 성공하면 UX 2/3 통과다. 2/3 미만은 제품성 점수와 무관하게 Fail이다. 결제 의향·한국 시장 크기·경쟁 제품은 UX 과업에 넣지 않는다.

## 3. 진짜 하드 실패 여덟 가지

다음 중 하나만 해당해도 Fail이다.

1. 범위를 쪼개도 1인 빌더가 4주 안에 핵심 가치를 만들 수 없다.
2. 필요한 입력에 현실적으로 접근할 수 없다.
3. 의료·법률·세무·품질의 위험한 판정이 핵심 결과다.
4. 같은 포트폴리오 안에서 타겟·순간·입력·즉시 결과가 사실상 완전히 같다.
5. 가치가 생기려면 문제 발생 전에 설치·기록해야 하지만, 그 사전 순간에 제휴 없이 사용자를 만날 경로가 없다.
6. OS·마켓플레이스·결제사처럼 **플랫폼 주인이 같은 핵심 결과를 기본 기능으로 제공**하고, 제3자 MVP는 권한이 부족해 체크리스트나 수동 재확인만 더할 수 있다.
7. 파일이나 메모를 범용 AI에 한 번 넣어 받는 일회성 요약·비교가 전부이며, 전용 제품만의 누적 상태·실행·배포 이점이 없다.
8. 핵심 가치가 생기려면 결제하지 않는 제3자가 새 링크를 열고 확인해야 하지만, 사용자가 그 행동을 강제하거나 제품이 기존 업무 채널 안에 들어갈 방법이 없다.

경쟁 제품, 낮은 사용 빈도, 불확실한 결제는 자동 Fail이 아니다. 아래 점수와 실제 실험으로 판단한다.

### 3.1 플랫폼 주인 우위 게이트

일반 경쟁 제품과 플랫폼 기본 해결을 구분한다. 다음 네 조건을 모두 만족하면 사용자 검토 전에 Fail 처리한다.

1. 사용자가 원하는 핵심 결과가 OS 설정·기기 설정·공식 이전·백업·초기화·정산 화면 안에 이미 있다.
2. 사용자는 새 앱 설치나 같은 데이터 재입력 없이 그 결과를 무료로 얻는다.
3. 플랫폼 주인은 원본 상태 확인 또는 실제 실행 권한을 갖지만 제3자 앱은 그 권한이 없다.
4. 제3자 MVP가 만들 수 있는 차이는 독립적인 새 결과가 아니라 체크리스트·스크린샷·수동 확인의 재포장뿐이다.

자동 Fail 예시는 `휴대폰 이전 누락 확인`, `OS 백업 복원 확인`, `사진 클라우드 백업 상태`, `수리 전 개인정보 보호 모드`, `공장 초기화 완료 확인`이다.

다음은 자동 Fail이 아니다.

- 서로 다른 두 주체의 데이터를 맞추는 결과: 주문 내역 + PG 정산 + 은행 입금
- 사용자가 합법적으로 꺼낼 수 있는 데이터에서 새 결과를 만드는 경우: GitHub PR + 이력서용 성과 증거
- 한 플랫폼이 전체를 볼 수 없는 횡단 결과: 여러 계정의 전화번호·2단계 인증 변경표

### 3.2 사용자 검토 전 시장 폭 게이트

제품성 Pass라도 `희소 사건 + 드문 후속 조건 + 특정 협상 상태`처럼 낮은 확률 조건이 세 개 이상 겹치면 독립 New 후보로 사용자에게 보여주지 않는다. 더 넓은 기존 그룹의 변형으로 Merge하거나, 실제 고의도 검색량·결제 증거가 생길 때까지 Hold한다.

예: `계약 종료 + 보증금 미반환 + 분할 반환 합의`만을 위한 별도 앱은 `이사반납` 또는 금전 약속 대조 카드로 Merge한다.

### 3.3 범용 AI 즉시 대체 게이트

다음 네 조건을 모두 만족하면 사용자 검토 전에 Fail 처리한다.

1. 입력이 문서·메모 1~2개다.
2. 결과가 요약·차이표·질문 목록 한 번으로 끝난다.
3. ChatGPT·Claude에 같은 입력을 넣으면 5분 안에 충분히 비슷한 결과가 나온다.
4. 제품이 이후 상태를 소유하거나, 상대의 제출을 받거나, 반복 실행을 자동화하지 않는다.

`AI를 더 잘 프롬프트한다`는 유일 주장이 아니다. 전용 제품으로 남으려면 적어도 하나는 필요하다: 반복 상태, 입력 수집 링크, 검증 가능한 결과 포맷, 다음 행동 실행, 고유 배포 채널.

### 3.4 비결제 상대방 협조 게이트

결제자는 쓰고 싶어도 기사·집주인·고객·업체가 새 링크를 눌러야만 결과가 완성되는 제품은 기본 Fail이다. 다음 중 하나가 있을 때만 예외로 둔다.

- 상대방도 즉시 얻는 이익이 분명하다.
- 계약·결제·납품처럼 확인하지 않으면 기존 업무가 진행되지 않는다.
- 이메일·메신저·결제 화면 등 이미 쓰는 채널에 별도 가입 없이 들어간다.

## 4. 제품성 공통 점수 — 16점

각 항목은 0/1/2점이다.

| 항목 | 0점 | 1점 | 2점 |
|---|---|---|---|
| 타겟·순간 | 누구나·언제나 | 타겟 또는 순간만 구체적 | 둘 다 구체적 |
| 통증·욕망 | 없어도 됨 | 편의·가벼운 욕망 | 돈·기회·불안·성취가 큼 |
| 수요 증거 | 추정뿐 | 인접 행동·경쟁 제품 | 같은 행동에 실제 결제 |
| 유일 주장 | 기능 목록 | 타겟/순간의 한 끗 | 한 문장으로 유일 주장 가능 |
| 결제 위치 | 나중에 수익화 | 광고·제휴·간접 결제 | 결과 직전/직후 직접 결제 |
| 리텐션 정직 | 반복·단건 모두 불명 | 반복 또는 단건 중 하나만 가능 | 반복 루프 또는 단건 결제가 명확 |
| 첫 고객 | 불특정 대중 | 검색·커뮤니티 한 곳 | 구체 채널과 첫 20명 경로 |
| MVP 절제 | 4주 초과 | 축소하면 가능 | 1~3화면·입력 1개·결과 1개 |

- **Pass:** 12~16점
- **Hold:** 9~11점
- **Fail:** 0~8점

점수 Pass는 실제 매출 검증이 아니라 사용자에게 대표 카드를 보여줄 자격이다. 앱 그룹 추가는 사용자 검증 뒤에만 한다.

## 5. 사용 순간 접근성 하드 게이트

제품성 점수가 높아도 아래 게이트를 별도로 통과해야 한다. 특히 여행·수리·사고·환불처럼 저빈도 사건에만 쓰는 B2C는 `첫 고객` 점수 하나로 유입 위험을 덮지 않는다.

### 5.1 두 가지 선행 질문

1. **문제가 보인 뒤 시작해도 되는가?** 필요한 사진·문서·기록을 문제 발생 후에 모아도 핵심 결과가 완성되는지 확인한다.
2. **정확한 순간의 첫 20명에게 제휴 없이 도달할 수 있는가?** 구체적인 검색어, 이미 모인 커뮤니티, 반복 업무 채널 중 하나가 있어야 한다.

### 5.2 판정

| Moment | 판정 기준 | 최종 상태 상한 |
|---:|---|---|
| **2** | 문제 발생 후 시작해도 되거나, 사전 준비 순간에 명확한 검색·커뮤니티 수요가 있다. 첫 20명 모집 문장을 쓸 수 있다. | 제품성 점수대로 판정 |
| **1** | 타겟 커뮤니티는 있으나 정확한 순간 포착이 어렵거나, 현장에서 설치·학습해야 해 전환 마찰이 크다. | **Hold** |
| **0** | 문제 전에 기록해야 하지만 사용자는 그 필요를 아직 모르고, 정확한 순간은 업체·플랫폼이 통제하며 제휴 없이는 도달할 수 없다. | **Fail** |

`검색으로 찾을 수 있다`만으로는 2점이 아니다. 실제 검색 문구와 문제 발생 전후 중 언제 노출되는지까지 적어야 한다. `광고하면 된다`, `여행 커뮤니티`, `앱스토어`처럼 범위만 말하면 1점 이하로 본다.

### 5.3 저빈도 사건형 추가 규칙

- 문제 발생 **후** 바로 시작해도 완전한 결과가 나오면 사건당 결제가 가능하다.
- 문제 발생 **전** 사진·기록이 필수라면, 예약·구매·신청처럼 사용자가 스스로 검색하는 사전 행동이 있어야 한다.
- 호텔 체크인 카운터, 렌터카 인수장, 수하물 벨트처럼 급한 현장에서 처음 발견하고 설치해야 한다면 기본값은 Hold다.
- 업체의 예약 완료 메시지·확인 링크·QR에 들어가야만 자연스럽게 쓰이는 제품은 독립 B2C가 아니라 제휴형 B2B2C다.
- `Moment 0`인 독립 제품은 기능이 유용해도 포트폴리오 New Pass로 세지 않는다.

## 6. B2C 수익모델

B2C는 월 구독만 인정하지 않는다.

- 사건당 결제: 여행·이사·수리·환불·청구 1,900~4,900원
- 기간 패스: 시험·여행·이직·이사 7~30일
- 평생 구매: 설치형 유틸리티
- 결과 잠금: 분석·비교·카드·리포트 공개
- 광고·제휴: 검색·쇼핑·여행처럼 구매 행동과 붙는 경우
- 구독: 매주 또는 매달 같은 결과를 다시 받는 경우

낮은 빈도라도 한 번의 불안·기회·손실이 충분하면 Pass할 수 있다.

## 7. 1인 사업자 B2B 범위

다음을 모두 만족하는 마이크로 SaaS만 허용한다.

1. 대표자 한 명이 영업 미팅과 조직 결재 없이 결제한다.
2. 초기 사용자는 1~5명이고 복잡한 권한이 없다.
3. 외부 플랫폼 API 없이 수동 입력·CSV·공유 링크로 시작한다.
4. 결과가 매출 회수, 비용 방어, 고객 유지, 운영시간 절약 중 하나와 연결된다.
5. 첫 고객을 강사·프리랜서·크리에이터·소규모 판매자 커뮤니티에서 직접 찾을 수 있다.

## 8. 경쟁과 유사성

- 해외 원본 유사도 0.80~1.00은 허용한다. 동일 메커니즘도 가능하다.
- 국내 경쟁 제품의 존재는 수요 증거다. 경쟁사가 있다는 이유만으로 탈락시키지 않는다.
- 기존 제품과 같은 기능이어도 더 좁은 타겟·더 절박한 순간·더 작은 가격·더 쉬운 입력 중 하나가 분명하면 통과 가능하다.
- 포트폴리오 내부 유사도 0.80~0.95는 다른 타겟·순간·결과가 있으면 허용한다.
- 0.95 이상은 자동 실패가 아니라 수동 Merge 검토다. 타겟·순간·입력·결과까지 같을 때만 한 그룹으로 합친다.

## 9. 제목·UVP

제목과 첫 두 문장은 다음을 보여야 한다.

1. `당신`이 지금 잃는 돈·시간·기회 또는 원하는 감정
2. 사용 전과 후의 대조
3. 숫자·파일·사진·날짜처럼 보이는 실체
4. 최고가 아니라 하나의 유일 주장

좋은 예: `결제한 수강생 30명 중, 7일째에도 첫 결과가 없는 사람만 보여드립니다.`

## 10. 상태 정의

| 상태 | 의미 |
|---|---|
| New Pass | 고품질이며 새 그룹 후보. 사용자 검증 대상으로 올림 |
| Merge | 제품성은 있으나 기존 그룹의 변형. 기존 그룹 카드 재료로 흡수 |
| Hold | 실제 가격·입력·사용 빈도 또는 Moment 1의 유입 경로를 먼저 확인 |
| Fail | UX 하드 실패, 위험, 입력 접근 불가, Moment 0, 8점 이하 |
| Custom Reserve | 일반 포트폴리오에서는 Hold/Fail이지만 도메인 지식·기존 고객·제휴 채널이 있는 빌더용 재료로 별도 보관 |

`Custom Reserve`는 최종 품질 판정이 아니라 보관 위치다. New Pass 수와 100개 목표에는 포함하지 않으며, 실제 채널을 가진 사용자의 커스텀 아이디어를 만들 때만 다시 심사한다.
