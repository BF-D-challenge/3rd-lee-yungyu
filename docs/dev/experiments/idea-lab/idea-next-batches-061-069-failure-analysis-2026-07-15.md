# Candidate 배치 061~069 실패 원인 분석

## 결론

- 서로 겹치지 않는 원본 481개를 스트레스 3조합 → Latin-9 → 전체 27조합 순서로 검사했다.
- 단계별 판정 행을 합치면 2,190행이며 pass 518, review 612, fail 1,060이다.
- 전체 27조합까지 간 원본 3개도 27/27을 통과하지 못했다.
- 그러나 이 결과를 그대로 원본 품질 0점으로 해석하면 안 된다. 기존 파이프라인이 사용자가 보는 한국어 결과 제목이 아니라 `JumpEye`, `V9九宫格` 같은 해외 제품명을 제목으로 검사한 구조적 오류가 확인됐다.
- 기존 `risk_flags`도 같은 의미를 `title_unclear`, `title_ambiguity`, `opaque_title`, `제목불명확`처럼 제각각 기록해 원인 집계가 흔들렸다.

## 구체적인 예시

### 1. 검수 입력의 제목 오류

- 원본: `JumpEye`
- 실제 조합: 농구 코치가 점프 영상을 넣으면 점프 높이를 계산
- 기존 판정: “제목 JumpEye만으로 문제와 결과가 드러나지 않는다.”
- 문제: 앱에서 사용자에게 보여줄 제목은 원본 브랜드가 아니라 `농구 점프 영상 높이 계산` 같은 결과 제목이어야 한다.
- 수정: 카드 초안에서 `resultTitle`을 필수로 만들고 검수기는 그 값을 제목으로 평가한다. 해외 원본명은 근거 확인용 `source_name`으로만 전달한다.

### 2. 결제자와 순간 불일치

- 결제자: 식단 사진을 기록하는 체중 감량 직장인
- 순간: 아이의 성장 사진에 날짜를 남길 때
- 결과: 아이 성장 사진 날짜 워터마크
- 판정: 결제자와 필요한 순간이 서로 다른 사람의 문제다.
- 수정: 결제자 3개와 순간 3개를 독립 문장으로 생성하지 않고, 9개 payer×moment 쌍이 모두 자연스러운지 초안 단계에서 확인한다.

### 3. 원본 메커니즘 이탈

- 원본: 위치로 달·태양 정보를 계산해 낚시 시간을 보여줌
- 한 끗: 조수 데이터까지 넣은 바다낚시 시간표
- 판정: 원본에 없던 조수 데이터와 새 외부 의존성을 추가했다.
- 수정: `smallestBuild`에 입력 1개·처리 1회·결과 1개를 명시하고 원본에 없는 데이터나 기능을 금지한다.

### 4. 사전 기록 필요

- 원본: 프리랜서 작업 시간과 수익 추적
- 필요한 순간: 월말 외주 수익 정산
- 판정: 월말에 결과를 받으려면 작업할 때마다 미리 기록했어야 한다.
- 수정: 문제 발생 뒤 텍스트·URL·파일·사진·단일 기기 권한 중 하나를 바로 넣어 결과를 받을 수 없는 원본은 재작성 대상이 아니라 탈락으로 유지한다.

## 다음 파이프라인 변경

1. 카드 초안을 문자열 3개씩이 아니라 앱에 가까운 구조로 만든다.
   - payer: `value`, `detail`
   - moment: `value`, `detail`
   - twist: `value`, `detail`, `resultTitle`, `smallestBuild`
2. 검수기는 해외 제품명과 한국어 결과 제목을 구분한다.
3. `risk_flags`를 고정된 코드 목록으로 제한한다.
4. 기존 Latin 결과에서 가능성이 높은 review 원본 20~30개만 새 구조로 다시 작성한다.
5. 새 카드로 스트레스 3 → Latin-9 → 전체 27조합을 다시 검사한다.
6. 27/27 통과 전에는 `sample-data.ts`를 수정하지 않는다.
