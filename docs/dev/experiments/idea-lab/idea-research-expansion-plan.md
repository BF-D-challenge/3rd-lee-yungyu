# 아이디어 리서치 확장 실행 계획

작성일: 2026-07-12  
목표: 원본 리서치 8,406건을 추적 가능한 방식으로 탐색해 **사용자 승인 원본 그룹 100개**를 확보한다.  
최종 조합 목표: 원본 100개 × 원본별 27조합 = **2,700조합**  
앱 데이터 반영 원칙: 대표 카드 사용자 승인 전에는 추가하지 않는다.

## 1. 핵심 전략

8,406건을 모두 같은 깊이로 읽지 않는다. 다음 퍼널을 50개 단위로 반복한다.

```text
원본 50개
→ 자동 하드게이트 통과 약 25개
→ 한국 행동·경쟁·유입 상세검사 15개
→ 공통 평가표+UX 통과 카드 최대 10개
→ 사용자 검증
→ 최종 승인 목표 3~5개
```

10개를 채우기 위해 품질이 낮은 후보를 올리지 않는다. 최종 숫자보다 승인율과 실제 매력이 우선이다.

## 2. 현재 작업 먼저 마감

현재 재심사 후보 70개의 사용 순간 접근성 검사를 완료했고, 후속 플랫폼 주인 우위·시장 폭 사전 게이트까지 적용했다.

- Moment 검사 통과: 62개
- 사전 게이트 제거: 6개 — 플랫폼 주인 우위 Fail 5 / 시장 폭 Merge 1
- 현재 일반 사용자 검증 후보: 0개 — 62개 전수 판정 완료
- Custom Reserve 이동: 8개
- 미검사: 0개

일반 검증 후보 62개를 모두 판정했고, Experiment Pass 4개·Merge 11개·Custom Reserve 5개·Fail 42개로 정리했다. 원본 8,406건 커버리지 장부도 생성했다.

| Moment | 처리 |
|---:|---|
| 2 | 일반 사용자 검증 후보로 유지 |
| 1 | 일반 포트폴리오에서는 Hold, 전문가용 Custom Reserve 검토 |
| 0 | 독립 제품으로 Fail, 도메인·채널 보유자용 Custom Reserve 검토 |

전체 결과 정본은 [사용 순간 접근성 전체 감사 — 70개](./idea-moment-access-audit-70.md)에서 관리한다.

### 3.4 상태 정의

| 상태 | 의미 |
|---|---|
| `unseen` | 아직 검사하지 않은 원본 |
| `in_app_unverified` | 현재 앱에 있지만 사용자 승인·품질 통과를 별도로 확인할 원본 |
| `auto_reject` | 기계적 하드게이트에서 탈락 |

## 4. 신규 원본 50개 회차 구성

한 회차에서 아직 보지 않은 원본 50개를 다음 비율로 가져온다.

| 소스 | 수 | 선택 기준 |
|---|---:|---|
| TrustMRR | 20 | 매출·설명이 있고 입력→처리→결과가 구체적인 원본 |
| 한국 App Store | 20 | 한국 순위·평가·리뷰에서 반복 행동이 보이는 앱 |
| Chrome Web Store | 10 | 설치형 또는 웹 MVP를 1인이 작게 만들 수 있는 확장 프로그램 |
| **합계** | **50** |  |

초기 3회 결과를 보고 소스 비중을 조정한다. B2C 통과가 부족하면 App Store 비중을 늘리고, 기업·전문가용 Custom Reserve가 과도하면 TrustMRR 비중을 줄인다.

## 5. 50개 선별 게이트

아래 순서대로 검사한다.

1. 구체적인 타겟·문제·결과가 있는가
2. 한국에서 실제로 반복하거나 검색하는 행동인가
3. 지금 잃는 돈·시간·기회·불안을 해결하는가
4. 기존 무료 도구보다 즉시 결과가 분명한가
5. 제휴 없이 필요한 순간의 사용자를 만나는 `Moment 2`인가
6. 1인 빌더가 4주 안에 핵심 MVP를 만들 수 있는가
7. 제품성 공통 점수가 12점 이상인가
8. 고정 P1/P2/P3 UX가 2/3 이상인가
9. 기존 그룹과 같다면 새 그룹이 아니라 Merge할 수 있는가
10. 사람이 봤을 때 실제로 써보고 싶은가

### 자동 탈락 또는 분리 대상

- 자체 모델 학습·대규모 크롤링·복수 외부 연동이 핵심
- 의료·법률·세무의 위험한 판정이 핵심 결과
- 양면 마켓플레이스나 엔터프라이즈 영업 없이는 시작 불가
- 필요한 입력을 현실적으로 가져올 수 없음
- 문제 발생 전 기록이 필수지만 제휴 없이 그 순간의 사용자를 만날 수 없음
- 기존 그룹과 타겟·순간·입력·결과가 사실상 동일

기능은 유용하지만 도메인 전문가만 유통할 수 있는 후보는 삭제하지 않고 `Custom Reserve`로 보낸다.

## 6. 사용자 검증 방식

사용자에게 원본 50개를 모두 보여주지 않는다.

### 6.1 1차 — 10개 요약 검토

상위 후보 최대 10개를 다음 정보와 함께 제시한다.

- 제목과 유일 주장
- 구체적 타겟
- 사용 순간
- 첫 사용자 도달 경로
- 예상 결제 위치와 가격
- 가장 작은 MVP
- 기존 앱과 한 가지 차이

사용자는 각 후보를 Yes/No로 빠르게 선별한다.

### 6.2 2차 — Yes 후보 상세 검토

Yes 후보만 한 개씩 전체 카드로 보여준다.

```text
제목·UVP
통증 미니드라마
타겟
기존 앱 vs 차별점
전체 플로우
가격·CTA
```

최종 Yes를 받은 후보만 `approved` 상태로 바꾼다.

### 6.3 실제 제작 후보 검증

사용자의 Yes는 포트폴리오 매력 검증이다. 실제 제작 우선순위 상위 3개는 별도로 검증한다.

- 타겟 사용자 5명에게 현재 어떻게 문제를 해결하는지 질문
- 제품 아이디어를 먼저 설명하지 않고 현재 행동·빈도·비용 확인
- 가능하면 실제 작업 화면이나 문서를 보여달라고 요청
- “쓸 것인가”보다 결제 버튼·사전 신청·파일 제출 같은 행동 확인

## 7. 승인된 뒤에만 27조합 생성

승인 전에는 원본별 조합을 만들지 않는다. 승인된 원본만 다음 구조로 확장한다.

```text
타겟 3 × 순간 3 × 한 끗 3 = 원본당 27조합
```

조합 생성 뒤 다음을 검사한다.

1. 해외 원본 충실도 임베딩 0.80 이상
2. 포트폴리오 내부 유사도 0.95 이상 수동 Merge 검토
3. 경계 조합의 타겟·순간 모순 여부
4. 각 조합의 1인 MVP 가능성
5. 제목·UVP 뉴로마케팅 품질
6. 대표 카드 사용자 승인과의 일관성

해외 원본과 0.80~1.00 유사한 것은 허용한다. 포트폴리오 내부에서도 타겟·순간·즉시 결과가 다르면 높은 유사도를 자동 탈락시키지 않는다.

## 8. 회차 중단·전환 기준

| 관찰 결과 | 다음 조치 |
|---|---|
| 50개 회차의 최종 승인 3개 미만이 2회 연속 | 같은 소스·카테고리 중단 |
| 사용자에게 보여준 10개 중 70% 이상 탈락 | 생성량이 아니라 선별·카피 기준 개선 |
| Merge 비율 40% 이상 | 포화된 카테고리를 멈추고 새 행동 영역으로 이동 |
| Custom Reserve 비율 30% 이상 | 제휴형 소스를 줄이고 한국 App Store 비중 확대 |
| 최종 승인 5개 이상 | 같은 소스 구성으로 다음 회차 진행 |

처음에는 신규 원본 **3회차, 총 150개까지만** 실행한다. 이후 실제 승인율을 이용해 남은 회차를 계산한다.

```text
예상 남은 회차 =
ceil((100 - 현재 사용자 승인 그룹 수) / 최근 3회 평균 승인 수)
```

예시: 기존 32개에 현재 후보 중 30개가 최종 승인되면 총 62개다. 신규 50개 회차당 평균 4개가 승인되면 38개를 더 확보하는 데 약 10회가 필요하다.

## 9. 100개에 포함되는 것

다음 조건을 모두 만족한 원본 그룹만 100개에 포함한다.

- 공통 평가표 통과
- UX 2/3 이상
- Moment 2
- 제품성 12점 이상
- 1인 MVP 가능
- 기존 그룹과 구분되거나 올바르게 Merge됨
- 사용자 최종 Yes

`Merge`, `Hold`, `Fail`, `Custom Reserve`는 100개에 포함하지 않는다.

## 10. 실행 순서

1. ~~남은 20개 Moment 검사~~ — 완료
2. ~~현재 70개 후보 전체 상태 확정~~ — 일반 검증 62 / Custom Reserve 8
3. ~~원본 커버리지 장부 생성~~ — 8,406건, `unseen 8,376 / in_app_unverified 30`
4. ~~현재 일반 검증 후보를 사용자 검증~~ — 62개 전수 완료, 실험 후보 4개
5. 사용자 승인 그룹 수를 기준선으로 확정
6. 신규 원본 50개 파일럿 1회
7. 상위 10개 요약 검토와 Yes 후보 상세 검증
8. 같은 방식으로 총 3회차까지 실행
9. 실제 승인율로 남은 회차 재계산
10. 승인된 원본만 앱 데이터와 27조합에 추가

## 11. 회차별 산출물

각 50개 회차는 다음 파일을 남긴다.

- 원본 50개 목록과 선택 근거
- 자동 탈락 목록과 표준 탈락 코드
- 상세 조사 15개 기록
- 사용자에게 보여줄 최대 10개 카드
- 사용자 Yes/No와 상세 검증 결과
- Merge·Custom Reserve 이동 기록
- 승인된 원본과 다음 회차에 반영할 학습

이 방식의 핵심은 원본 수집, 제품성 평가, 사용자 취향, 실제 시장 검증을 하나의 점수로 섞지 않는 것이다. 원본 데이터는 발굴 재료이고, 최종 포트폴리오 품질은 사용자 승인과 실제 행동 증거로 결정한다.
