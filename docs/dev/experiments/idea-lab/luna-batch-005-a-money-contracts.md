### A03 보험 보완요청 타임라인

- Hook: “보험사에 서류를 보냈는데 또 무엇을 달라는지 헷갈리는 순간, 사건 하나를 제출 증명까지 닫는다.”
- 구체 타겟: 실손·자동차·정액보험을 연 2회 이상 청구하거나 한 건의 예상 지급액이 5만원을 넘는 개인.
- 기존 앱과 차이: 보험금의 지급 판단이나 청구 접수를 대체하지 않고, 문자·전화 후속·메일·파일을 한 사건의 요청일/제출일/재요청 상태로 소유한다.
- 전체 플로우: 보완요청 캡처 업로드 → 항목 1개씩 체크 → 파일 첨부와 제출일 기록 → 보험사 답변일 리마인드 → 제출 증명 PDF.
- 4주보다 작은 MVP: 사건 1건, 항목 10개, 캡처/파일 첨부, 상태 4개, 날짜 알림, PDF export의 3화면.
- 결제 시점·가격 가설: 첫 사건의 보완팩 PDF를 내보낼 때 3,900원 사건 패스; 보험대리인용 월 4,900원은 후속 검증.

- 구체 타겟: 매달 3곳 이상에서 용역비를 받는 프리랜서 디자이너·개발자·번역가.
- 기존 앱과 차이: 세금을 계산하거나 홈택스를 복제하지 않고, 의뢰처별 지급월·요청일·수령 파일·확인 상태와 요청문 발송 기록을 소유한다.
- 전체 플로우: 의뢰처와 지급내역 입력 → 영수증 파일 추가 → 미수령 자동 표시 → 요청문 복사/발송 기록 → 연말 ZIP export.
- 4주보다 작은 MVP: 의뢰처 10곳, 지급월 12개, 파일 첨부, 수령 상태, 요청문, ZIP/CSV export.
- 결제 시점·가격 가설: 연말 export 직전에 4,900원 1년 보관 패스; 월 구독은 반복 사용 확인 뒤 보류.

### A14 검수 기준표

- Hook: “납품 링크를 보냈다는 사실이 아니라, 계약의 검수 항목이 승인됐는지로 잔금 준비를 판단한다.”
- 구체 타겟: 월 3건 이상 디자인·영상·개발 납품을 하는 1인 프리랜서.
- 기존 앱과 차이: 파일 버전 자체를 관리하는 제품이 아니라 계약서에서 합의한 검수 항목, 고객의 항목별 승인, 수정요청 상태를 한 번에 보존한다.
- 전체 플로우: 계약 페이지와 납품 링크 입력 → 검수 항목 3~10개 등록 → 고객 확인 링크 → 항목별 승인/수정요청 → 잔금 준비 상태와 로그 export.
- 4주보다 작은 MVP: 프로젝트 1건, 링크 1개, 검수 항목 5개, 고객 확인 링크, 상태 로그, PDF export.
- 결제 시점·가격 가설: 고객 확인 링크를 발송할 때 프로젝트 1건 2,900원; 월 10건 9,900원은 반복 납품자 대상.

### A22 월세 인상 통지 확인

- Hook: “다음 달 월세가 올랐다는 말만 남지 않게, 통지·협의·합의·미확인을 한 줄로 구분한다.”
- 구체 타겟: 갱신·재계약을 앞둔 원룸/오피스텔 세입자와 가족 대신 계약을 확인하는 보호자.
- 기존 앱과 차이: 인상 가능 여부를 법률 판정하지 않고, 기존 계약 금액·통지일·새 금액·양측 확인 상태와 실제 입금액을 사건으로 묶는다.
- 전체 플로우: 계약서와 통지 캡처 입력 → 새 금액/적용일 확인 → 집주인 확인 링크 → 합의/미확인 상태 → 첫 납부 후 실제 금액 기록.
- 4주보다 작은 MVP: 계약 1건, 통지 캡처, 금액 2개, 상태 4개, 확인 링크, PDF export.
- 결제 시점·가격 가설: 재계약 문서를 export할 때 2,900원 사건 패스; 연 1회 이상 계약을 관리하는 보호자에게 4,900원 1년 패스.

## Hold 상세 브리프

### A08 종소세 증빙 반려 큐

- Hook: “신고가 끝난 뒤 ‘이 경비는 다시 내라’는 보완통지에서 필요한 증거 한 건만 찾는다.”
- 구체 타겟: 연 1회 종합소득세 신고에서 경비 증빙이 많고 보완금액이 5만원 이상인 프리랜서.
- 기존 앱과 차이: 신고·세액 계산을 하지 않고, 보완통지의 사유와 영수증·거래처·재제출 상태를 연결한다.
- 전체 플로우: 보완통지 입력 → 반려 항목 선택 → 원본 증빙 첨부 → 세무사/홈택스 제출 기록 → 처리 결과 보관.
- 4주보다 작은 MVP: 통지 1건, 항목 10개, 파일 첨부, 상태 4개, 재제출 PDF.
- 결제 시점·가격 가설: 보완팩 export 시 4,900원 사건 패스. Hold 이유는 실제 보완통지 빈도와 세무사 대체 대비 결제 검증 부족.

### A15 수정횟수 계산표

- Hook: “수정 요청이 계약 범위를 넘었는지, 버전 수와 조항을 같이 보며 한눈에 확인한다.”
- 구체 타겟: 월간 콘텐츠·디자인 납품을 반복하고 고객별 수정 조건이 다른 프리랜서.
- 기존 앱과 차이: 작업관리 전체가 아니라 계약서의 수정 허용 횟수, 요청별 버전, 초과 여부, 추가견적 확인만 소유한다.
- 전체 플로우: 수정 조항 입력 → 요청 메시지와 버전 연결 → 포함/초과 판정 → 고객 확인 → 추가견적 로그 export.
- 4주보다 작은 MVP: 프로젝트 1건, 수정횟수 3개, 요청 10개, 링크 첨부, 상태 로그.
- 결제 시점·가격 가설: 초과 요청에 대한 추가견적 PDF export 때 2,900원. Hold 이유는 배치 003 `범위밖` 및 일반 프로젝트 관리도구와 경계가 가깝다.

## Fail 이유 — 각 한 문장

- A01: 보험사 앱이 청구 결과를 소유하고 개인 앱은 보완 요청을 다시 입력하게 되어 대안·결제 우위가 약하다.
- A02: 보험사·실손 청구 플랫폼이 동일한 서류 묶음 결과를 직접 제공한다.
- A04: 보험사 지급내역과 은행 입금내역을 사용자가 다시 입력해야 하며 차액 결과가 단건이다.
- A05: 사고 처리 순서와 심사는 보험사가 소유하고 별도 앱의 결제 결과가 없다.
- A06: 기관 절차의 일회성 보완이고 공단 제출 결과를 제품이 소유할 수 없다.
- A07: 병원·정부24·보험사가 발급과 청구를 이미 해결한다.
- A09: 카드사·가계부·회계앱의 분류 기능과 결제 결과가 겹친다.
- A11: 홈택스·회계앱·세무사가 누락 증빙을 더 강하게 관리한다.
- A12: 환율·플랫폼별 파서와 세무 정확성으로 4주 MVP를 넘고 세무사 대안이 강하다.
- A13: 전자세금계산서 수취 확인은 홈택스가 법적 기준과 결과를 소유한다.
- A16: 반환 가능성의 법률 판정이 핵심이 되어 법률 대안과 충돌한다.
- A17: 배치 002 `급여차이`와 금액 입력→차액 결과가 사실상 동일하다.
- A18: 공동 송금·확인 자체는 메신저와 계좌이체로 충분하고 반복 결제 결과가 약하다.
- A19: 배치 003 `라이더진짜시급`·배치 004 `플랫폼 정산 차이`와 입력·결과가 중복된다.
- A20: 영수증·잔액 표는 스프레드시트와 회계앱으로 대체되며 회수 손실이 직접적이지 않다.
- A21: HUG가 보증 사고 접수·심사 결과를 소유하고 연 1회성이라 독립 결제가 약하다.
- A23: 퇴거 시점의 일회성 반환이며 관리사무소·임대인이 절차를 소유한다.
- A24: 공식 기준과 무료 계산·중개사 확인으로 충분하고 오류 시 법적 책임이 크다.
- A25: 배치 002 `하자증거`와 배치 004 `수리비 선지급 장부`의 사진·요청·합의/상환 증거 흐름이 중복된다.
