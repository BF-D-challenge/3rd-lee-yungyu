## 2. 루트 교차 감사 기준

모든 후보를 제목이 아니라 `입력 → 제품이 소유하는 누적 상태 → 즉시 결과`로 다시 비교했다.

1. P1 신입·P2 중급·P3 숙련 중 두 명 이상이 `10초 이해/한국 실제 사용/결제 이유`를 모두 Yes로 평가해야 다음 게이트로 간다.
2. UX 2/3은 통과가 아니라 하드 게이트 진입권이다. 국내 플랫폼·OS·공공기관·전문 사업자가 핵심 결과를 소유하면 Fail이다.
3. 사용자가 아니라 강사·사업자·조직이 결제해야 성립하면 B2C 포트폴리오에서 Fail이다.
4. 기존 앱 32개 그룹과 배치 001~005, 동시 006~009의 입력·상태·결과가 겹치면 Fail이다. 필드나 타겟만 바꾼 80% 변형은 새 원본 그룹으로 세지 않는다.
5. 1인 바이브 코딩으로 4주 MVP가 불가능하거나 법률·의료·품질 판정, 대규모 연동, 양면 마켓플레이스가 필요하면 Fail이다.
6. Hold는 실제 사용자에게 가격을 먼저 보여주는 작은 실험을 할 가치가 있으나 아직 결제 근거가 없는 후보만 남긴다.

## 3. 최종 Hold 2개

### D06-12 호텔 객실 상태팩

- UVP: 체크인 직후 객실 하자와 프런트의 조치 약속을 사진 시각과 함께 한 사건으로 남긴다.
- 남긴 이유: 여행 중 보증금·책임 분쟁이라는 구체 손실이 있고, 사진·프런트 답변·조치 상태를 함께 묶는 결과가 명확하다.
- 아직 Pass가 아닌 이유: 예약 플랫폼 채팅과 기본 카메라보다 건당 결제할 우위가 검증되지 않았다.
- 최소 실험: 실제 여행객 10명에게 체크인 직전 `하자 증거 PDF 1,900원` 결제 화면을 보여주고, 최소 2건 선결제 후 4주 MVP에 진입한다.

### D08-02 백업 복원 시험

- UVP: “백업 완료” 표시 대신 실제로 열리는 시험 파일 한 개를 확인하고 다음 복원 시험일을 남긴다.
- 남긴 이유: 백업 서비스가 소유하지 않는 `사용자가 실제 복원에 성공한 상태`를 별도로 소유하며, 실패 시 손실이 크다.
- 아직 Pass가 아닌 이유: Backblaze·iCloud 등 기존 백업 서비스의 복원 기능과 별도 결제할 반복 가치가 검증되지 않았다.
- 최소 실험: 사진·문서 백업 사용자 10명에게 `월 2,900원 복원 점검` 선결제를 요청하고, 최소 2건 결제와 두 번째 점검 예약이 확인된 뒤 MVP에 진입한다.

## 5. 포화 해석과 다음 의사결정

- 배치 005의 루트 최종 Pass는 5/100이었지만, 이번 006~009는 **0/400**이다. 따라서 과거의 5% 통과율로 “몇 회 더 하면 된다”고 예측할 수 없다.
- 이번 결과는 UX 기준이 너무 엄격해서라기보다, 넓은 카테고리 조합이 이미 기존 후보의 타겟·필드 변형을 반복하기 시작했다는 신호다.
- 다음 단계는 새 100개 생성이 아니다. Hold 2개를 실제 선결제 화면으로 검증하고, 동시에 한국에서 반복 결제가 이미 일어나는 행동 원천을 새로 수집해야 한다.
- 합성 UX 평가는 선별용 가설이다. 실제 사용자 이해·사용 의도·결제를 대신하지 않으며, 앱 데이터 반영은 대표 카드 사용자 검증 이후에만 한다.
