## 발굴·리서치 규칙

- Firecrawl은 사용하지 않는다.
- 로컬 TrustMRR·App Store 자료로 발산하고, 최종 통과 가능 후보는 일반 웹 검색·Jina Reader·공식 페이지·앱스토어·인증된 Reddit CLI 중 가능한 수단으로 확인한다.
- 최소 3개의 신뢰 가능한 출처를 보고서 끝에 URL과 함께 기록한다. 수집 실패는 시장 부재로 해석하지 않는다.
- 해외 원본은 80~99% 또는 완전 동일해도 된다. 브랜드·로고·고유 카피·에셋은 복제하지 않는다.

## 제품 하드 게이트

1. 한국 타겟이 월 1회 이상 겪거나 예상 손실이 5만원 이상이다.
2. 한국의 공공기관·플랫폼·강한 무료 앱이 핵심 결과를 이미 해결하면 실패다.
3. ChatGPT·Claude 한 번 질문으로 끝나면 실패다. 상태·시간 누적·증거·반복 큐·양측 확인 중 하나를 제품이 소유해야 한다.
4. 혼자 바이브 코딩 가능한 4주 MVP다. 범위가 크면 핵심 입력 1개→처리 1회→결과 1개로 쪼갠다.
5. 은행 OAuth, 대규모 크롤링, 양면 마켓플레이스, 의료 진단, 법률 판정, 상시 백그라운드 수집이 필요하면 실패다.
6. 제목은 `누구의 어떤 손실/마찰을 어떤 즉시 결과로 바꾸는지` 10초 안에 보여준다. 추상 카테고리명과 `AI 생성기·추천`은 금지한다.

## UX 합성 평가

각 후보를 같은 ASCII 카드로 본다고 가정하고 다음 3개 경험층이 독립적으로 빠르게 스캔한다.

- P1 신입: 제목과 첫 결과만 보고 이해·진입 가능 여부
- P2 중급: 오류·확인·되돌리기와 반복 효율
- P3 숙련: 기본 도구 대비 우위·결제 결과·효율 천장

각 페르소나는 `10초 이해 / 한국 실제 사용 / 결제 이유`를 Y/N으로 남긴다. 세 항목 모두 Y인 페르소나가 2명 미만이면 즉시 Fail이다. UX 2/3 이상이어도 국내 대안·결제·MVP 하드 게이트 중 하나가 실패하면 최종 Fail이다. 합성 평가는 실제 사용자 검증이 아닌 선별 가설이다.

## 출력 형식

1. 정확히 25행 후보 표
   - ID
   - 제목 — 한 줄 UVP
   - 실제 입력 → 소유 상태 → 즉시 결과
   - 해외·한국 근거
   - P1/P2/P3 (`이해/한국/결제`)
   - UX 합의
   - 하드 게이트
   - 최종 Pass/Hold/Fail
2. Pass와 Hold만 상세 브리프
   - Hook
   - 구체 타겟
   - 기존 앱과 차이
   - 전체 플로우
   - 4주보다 작은 MVP
   - 결제 시점·가격 가설
3. Fail은 이유 한 문장
4. 출처 URL, 통과/보류/탈락 수, 기존 후보 중복 검사

## 파일 규칙

- 담당 출력 파일만 수정한다.
- 기존 보고서와 앱 데이터는 수정하지 않는다.
