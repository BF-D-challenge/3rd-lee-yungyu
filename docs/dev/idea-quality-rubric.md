# 아이디어 품질 루브릭 — 원본 그룹별 yes/no 평가

> 평가 단위 = `해외 원본 1개 + 타겟 3 + 순간 3 + 끗 3`으로 이루어진 원본 그룹 하나.
> 그룹 하나는 한 끗·두 끗을 포함해 54개 조합을 만든다. 현재 1,728장을 같은 질문으로 반복 평가하지 않고, 그룹 공통 평가 → 대표·경계·최악 조합 3장 평가 순서로 진행한다.

## 페르소나 (통제 3층)
- **P1 신입(막막형)**: 사이드프로젝트 처음. "이거 오늘 나도 만들 수 있겠다"가 판단 기준.
- **P2 중급(확신부족형)**: 아이디어는 많음. "이게 진짜 수요 있나·차별되나"가 기준.
- **P3 숙련(도메인전문·시간빈곤)**: 업계 눈. "유치하지 않나·근거 있나·바로 실행되나"가 기준.

## 평가 순서

### 1단계. 자동 그룹 게이트

1. 원본의 매출·사용량 근거가 존재한다.
2. 54개 조합이 모두 필수 데이터를 가진다.
3. 각 조합의 해외 원본 충실도가 0.80 이상이다. 0.99 초과도 허용한다.
4. `입력 1개 → 처리 1회 → 결과 1개`로 축소할 수 없는 조합을 표시한다.
5. 수요 근거에서 반복되는 **실제 입력**을 MVP가 받을 수 있는지 확인한다. 스크린샷 문제를 링크 붙여넣기로 검증하는 식으로 입력을 바꾸면 탈락이다. 단, 설치형 테스트 앱으로 권한 요청부터 실제 입력까지 구현할 수 있으면 허용하며 그 구현을 반나절~2일 범위에 포함해 판정한다.

#### 구조적 차별화 게이트

해외 원본을 80~99% 유지하거나 완전히 복제하는 것은 허용한다. 그러나 최종 포지셔닝이 `Claude에 프롬프트 한 번 입력` 또는 기존 경쟁사의 기능 하나로 대체되면 사용자 평가 전에 탈락시킨다.

1. **수요의 언어와 차별점을 분리한다.** `회의 AI`, `이세계`, `식단 코치` 같은 카테고리 이름은 수요 증거이지 차별점이 아니다.
2. **경쟁 구조를 하나 이상 뒤집는다.** 예: 상시 카탈로그 → 오늘 하나, 전체 녹취 → 녹취 없이 상호 확인, 무한 대화 → 턴 수가 있는 완결.
3. **핵심 약속을 시스템이 집행한다.** 프롬프트가 그렇게 말하도록 연출하는 것이 아니라 서버 상태·양쪽 확인·결정론 규칙처럼 사용자가 임의로 바꿀 수 없는 제품 동작이어야 한다.
4. **인컴번트 복제 테스트를 통과한다.** 경쟁사가 기능 하나를 추가해 그대로 복제할 수 있으면 차별화 실패다. 따라 하려면 기존 탐색·기록·과금·콘텐츠 공급 구조 중 하나를 스스로 약화해야 한다.
5. **범용 AI 대체 테스트를 통과한다.** Claude·ChatGPT에 입력 하나를 넣어 같은 결과를 받을 수 있다면 별도 제품이 아니다. 여러 사람 사이의 상태, 시간에 따른 누적, 검증된 데이터, 반복 실행 중 하나를 제품이 직접 소유해야 한다.

이 게이트는 사용자 취향 문항이 아니다. 경쟁 조사와 제품 구조를 근거로 제작자가 먼저 판정하며, 실패한 아이디어는 제목을 고치지 말고 제품 루프를 반대로 재설계하거나 폐기한다.

#### 한국 적용성·유용성 게이트

해외 매출이 있어도 한국에서 드물거나 이미 다른 서비스에 포함된 문제라면 공개하지 않는다. 아래 다섯 항목을 모두 통과해야 한다.

1. **한국의 실제 행동 근거가 있다.** 국내 앱 순위·검색·커뮤니티 VOC·플랫폼 사용행태 중 하나 이상에서 같은 문제를 확인한다. 해외 사용자가 있다는 사실만으로 대체하지 않는다.
2. **충분히 자주 발생한다.** 핵심 타겟이 적어도 월 1회 반복해서 겪는다. 일회성 문제는 예상 손실·지출이 5만원 이상이고 현재 해결이 불편할 때만 예외로 둔다.
3. **한국의 기본 대체수단보다 낫다.** 예식장·학원·플랫폼·가족이 이미 대신해주는 일을 별도 앱으로 다시 만들지 않는다.
4. **한국에서 결제할 결과가 보인다.** 해외 구독 가격을 그대로 복사하지 않고, 국내 사용자가 돈을 내는 결과·시점·가격 단위를 설명한다.
5. **시장 크기를 과장하지 않는다.** `사람이라면 누구나`, `학생`, `직장인`처럼 넓게 쓰지 말고 실제 반복 사용자 수가 충분한 한 집단을 제시한다.

이 게이트가 하나라도 No면 12문항 점수가 높아도 탈락이다. 10개 묶음은 통과 수가 부족하면 10개보다 적게 공개한다.

자동 게이트 실패는 사용자에게 취향 질문으로 넘기지 않고 데이터부터 수정한다.

### 2단계. 사용자 그룹 공통 평가

사용자는 개별 문장보다 그룹을 만드는 공통 재료를 Yes/No로 평가한다.

1. 원본의 핵심 가치가 한국에서도 필요해 보이는가?
2. 타겟 3명이 같은 핵심 문제를 반복해서 겪는가?
3. 필요한 순간 3개가 모두 실제 행동 직전의 절박한 장면인가?
4. 끗 3개가 원본 가치를 망가뜨리지 않는가?
5. 끗 두 개를 결합해도 하나의 제품 약속으로 이해되는가?
6. 가장 큰 조합도 1인 바이브 코딩 MVP로 쪼갤 수 있는가? **필수**
7. 타겟의 실제 입력을 다른 것으로 바꾸지 않고 외부 플랫폼·데이터 의존을 우회할 수 있는가? **필수**
8. 세 타겟 중 적어도 한 명은 돈을 낼 이유가 분명한가?
9. 그룹 안의 54개가 같은 말만 바꾼 카드가 아니라 사용 순간·결과 차이를 만드는가?
10. 이 그룹에서 다음 대표 아이디어를 더 보고 싶은가?

6번 또는 7번이 No면 그룹을 폐기하지 않고 핵심 아이디어를 더 작게 쪼개 재평가한다. 나머지는 8/10 이상을 그룹 통과 기준으로 사용한다.

### 3단계. 그룹당 샘플 3장 평가

- **대표 조합**: 그룹의 핵심 약속을 가장 잘 보여주는 카드
- **경계 조합**: 타겟·순간 정합성이 가장 약한 카드
- **최대 조합**: 두 끗을 붙여 범위가 가장 커진 카드

세 장은 아래 32개 세부 항목 전체가 아니라 `제목 이해·pain 정합성·결과 매력·결제 이유·1인 MVP` 5개만 평가한다. 하나라도 No면 그 조합에 참여한 타겟·순간·끗 재료를 수정하고 54개를 다시 생성한다.

### 4단계. 사용자 개별 카드 검증 — 고정 12문항

사용자에게 아이디어를 하나씩 보여줄 때는 아래 문장을 **단어 하나도 아이디어에 맞춰 바꾸지 않고**, 같은 순서로 반복한다. 아이디어별 사례·입력·결과는 평가 질문이 아니라 아이디어 브리프에만 쓴다.

#### 공통 품질 10문항

1. 제목과 한 줄 설명만으로 제품이 무엇을 하는지 이해되는가?
2. 돈을 낼 사람이 구체적으로 한 종류로 보이는가?
3. 해결하려는 문제가 실제 행동 직전의 구체적인 순간으로 보이는가?
4. 사용자가 현재 이 문제를 해결하기 위해 이미 시간·돈·노력을 쓰고 있다고 믿을 수 있는가?
5. 해외 원본의 매출·사용량 근거가 이 문제의 실수요 증거로 충분한가?
6. 누가 어떤 반복 가치에 돈을 낼지 명확한가?
7. 핵심 가치가 `입력 1개 → 처리 1회 → 결과 1개`로 설명되는가? **필수**
8. 1명이 바이브 코딩으로 반나절~2일 안에 실제 작동하는 MVP를 만들 수 있는가? **필수**
9. 타겟이 실제로 쓰는 입력을 바꾸지 않고, 직접 입력·관리형 API 1~2개 또는 설치형 테스트 앱의 단일 기기 권한으로 구현할 수 있는가? **필수**
10. 브랜드·카피·에셋을 복제하지 않고 안전하게 출시할 수 있는가? **필수**

#### 개인 선호 2문항 — 품질 점수와 분리

11. 나는 이 아이디어의 결과를 직접 확인해보고 싶은가?
12. 나는 이 아이디어를 다른 사람에게 보여주거나 추천하고 싶은가?

11~12번은 평가자가 실제 타겟이 아닐 수 있으므로 아이디어의 품질 합격 여부에 사용하지 않는다. 개인 선호·취향 학습 신호로만 별도 집계한다.

#### 개별 카드 판정

- 공통 품질: 1~10번 중 8개 이상 Yes
- 하드 게이트: 7~10번은 모두 Yes
- 개인 선호: 11~12번은 별도 `0~2`로 기록하고 합격점에 합산하지 않음
- 이전처럼 `회사 URL`, `근거 3줄` 등 아이디어별 단어를 질문 안에 넣은 평가는 공식 비교에서 제외

## yes/no 항목 (32 · 8각도)

### A. 구체성
1. 타깃이 구체적인가? ("혼자 뛰는 3개월차 러너" O / "러닝 좋아하는 사람" X)
2. pain이 1인칭 실감 문장인가? (추상 명사구 "벤치마크 부재" X)
3. 해결 메커니즘(형태)이 무엇을 하는지 명확한가? ("스트릭" X / "오늘 뛰었다 1탭 기록" O)
4. 제품이 한 문장으로 머릿속에 그려지는가?

### B. 실수요 증거
5. 비슷한 실제 제품/서비스가 존재하는가?
6. 그 제품이 매출 또는 사용량(순위·평점)으로 실수요가 증명됐는가?
7. 근거가 검증 가능한가(앵커 제품명·수치)?
8. "이거 나도 필요해(🔥)"를 부를 만한가?

### C. 1인 바이브 코딩 MVP인가 — 하드 게이트
9. 1명이 바이브 코딩으로 반나절~2일 안에 배포할 수 있는가?
10. 핵심 가치가 `입력 1개 → 처리 1회 → 결과 1개`로 쪼개졌는가?
11. 첫 버전이 1~3개 화면이며 로그인·결제·관리자·팀 권한 없이 작동하는가?
12. 대규모 크롤링·여러 OAuth·상시 작업·자체 모델 없이, 직접 입력·관리형 API 1~2개 또는 설치형 테스트 앱의 단일 기기 권한으로 만들 수 있는가?
13. 큰 원본을 그대로 만들라는 문구가 아니라, 핵심 세로 조각을 만드는 프롬프트인가?
14. 수요 근거의 실제 입력과 MVP 입력이 같은가? 네이티브 권한이 필요하면 설치형 테스트 앱의 권한 요청·선택·가져오기까지 핵심 플로우에 포함되어 있는가?

### D. 조합 정합성
15. pain ↔ 형태(해결책)가 실제로 맞물리는가? (선택마비×스트릭 X)
16. 씨앗 ↔ pain이 그 도메인에서 말이 되는가? (러닝×물류검수 X)
17. 장면(situation)이 자연스러운가(무관 노이즈 아님)?
18. 마음(psych)이 억지스럽지 않은가?
19. audience(소비자/메이커)가 일치하는가? (시청자×크리에이터 pain X)

### E. 해외 원본 충실도·적용 명료성
20. 해외 원본의 제품 메커니즘을 임베딩 기준 0.80~0.99 유지하는가? 0.99 초과의 사실상 동일 아이디어도 허용한다.
21. 기능이 원본과 같더라도 한국의 구체적 타겟·필요 순간·결과가 명확한가?
22. 차별화를 위한 불필요한 기능을 억지로 붙이지 않았는가?

### F. 언어·표현
23. 한국어 문장이 자연스러운가?
24. 조사가 정확한가(받침 처리)?
25. 제품명이 그럴듯한가(어색한 직역 X)?
26. 컨설팅 보고서 문체가 아닌가?
27. 영어 제품명·매출수치가 카드 표면에 노출 안 됐는가(앵커로만)?

### G. 감정·공유
28. 지인에게 보여줘도 창피하지 않은가?
29. 응원하고 싶은(긍정 반응 유발) 카드인가?
30. 스토리로 공유할 만큼 흥미로운가?

### H. 실행 위험
31. 제품 아이디어 복제와 별개로 브랜드명·로고·고유 카피·디자인 에셋을 복제하지 않고, 플랫폼 의존은 수동 입력 MVP로 우회했는가?
32. 금칙어(생성기·AI 추천·랜덤) 미노출인가?

## 집계·판정
- **UX 최우선 하드 게이트**: 고정 3페르소나 중 2명 미만이 `10초 이해·한국 실제 사용·결제 이유`를 모두 Yes로 평가하면 최종 실패다. 이전 사용자 승인, 구현 가능성, 시장 근거, 실험 의향으로 이 실패를 뒤집지 않는다.
- **2/3의 의미**: UX 2/3 이상은 다음 하드 게이트로 이동할 자격일 뿐 최종 통과를 보장하지 않는다. 2/3 미만은 즉시 실패하며, 수정한 시안으로 재평가해야만 다시 진입할 수 있다.
- **그룹 공통 판정**: 필수 2개 Yes + 전체 8/10 이상이어야 샘플 평가로 이동한다.
- **샘플 판정**: 대표·경계·최대 3장 모두 5/5여야 그룹을 최종 통과시킨다.
- **항목별 yes율**(통과 그룹×3페르소나): <70% = 시스템 약점 → 그 각도로 데이터/엔진 개선.
- **MVP 하드 게이트**: C의 No가 하나라도 있으면 공개하지 않는다. 아이디어를 폐기하지 말고 핵심 가치가 처음 발생하는 더 작은 세로 조각으로 분해한 뒤 다시 평가한다.
- **카드별 32개 세부 평가는 예외 감사용**: 그룹 샘플에서 실패 원인이 불분명할 때만 사용한다.
- **페르소나 편차**: 특정 층만 no → 그 층용 보정(신입=구체성/오늘만들기, 중급=차별성/수요, 숙련=근거/비유치).
- **개선 반복**: 약점 항목 상위 3개 고침 → 재채점. 최대 3라운드, yes율 목표 ≥85%.
