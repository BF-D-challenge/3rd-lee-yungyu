# 오늘 해볼까 - 모바일 최종 플로우 v10

> 상태: **IMPLEMENTED & VERIFIED**  
> 작성: 2026-07-10  
> 화면·모션 정본: [오늘 해볼까 - 모바일 최종 플로우](../dev/mobile-slot-candidates.html)  
> 대체 문서: [개선 계획 v9](오늘-해볼까-개선계획-v9.md)의 모바일 사용자 플로우

## 1. 목표

로그인한 사용자가 취향을 3개 이상 고르고, 현재 제품의 카드 비행·플립 물성을 유지한 채 완성 아이디어 카드 두 장을 90초 안에 얻는다. BF.D 멤버는 공유 링크에서 한 장을 고르고 긍정 칭찬을 남긴 뒤 자기 카드 두 장을 만드는 루프로 이어진다.

제품의 바이럴 완료점은 링크 클릭이나 가입이 아니라 **새 사용자가 자기 카드 두 장을 완성하고 새 BF.D 라운드를 공유한 순간**이다.

## 2. 고정 결정

1. 로그인은 모든 진입의 첫 행동이며 건너뛰지 않는다.
2. 첫 로그인 직후 취향 3-8개를 필수로 고른다. 별도 라우트가 아닌 `/`의 전체 화면 상태다.
3. 재방문 취향 편집만 게임판 위 모달 바텀시트로 연다.
4. 4개 축을 조합하지 않는다. 완성 아이디어 카드 A와 B를 한 장씩 직접 연다.
5. 카드 표면이 유일한 기본 뽑기 액션이다. 중앙 덱은 비행 출발점을 보여주는 장식이다.
6. BF.D 수신자는 로그인 후 두 카드 중 한 장과 긍정 칭찬 한 개를 고른다.
7. 칭찬 후 선택 카드의 취향을 이어받아 자기 카드 A/B 게임판으로 이동한다.
8. 핵심 플로우에서 별도 확정, 별도 발행, 4/4 칭찬 완성, 대시보드를 제외한다.

## 3. 최소 화면과 상태

| 화면 | 라우트 | 핵심 상태 | 완료 조건 |
|---|---|---|---|
| 로그인 | `/` | `idle -> authenticating -> returning -> success/error` | OAuth 복귀 맥락 보존 |
| 게임판 | `/` | `taste:first -> idleA -> revealingA -> idleB -> revealingB -> complete` | 카드 A/B 완성 및 공유 CTA 노출 |
| BF.D 응원 | `/vs/[slug]` | `authRequired -> ready -> chosen -> praiseSelected -> handoff` | 한 장 선택, 칭찬, 내 카드 시작 |

화면은 3개, 라우트는 2개다. 취향 첫 선택과 편집은 화면 수를 늘리지 않는 게임판 내부 상태다.

## 4. 화면 계약

### 4.1 로그인

- CTA는 `Google로 시작하기` 하나만 둔다.
- OAuth URL에 원래 라우트와 `slug`를 보존한다.
- 800ms 이내 인증은 로더를 생략하고, 그 이후에만 버튼 안에 지연 로더를 표시한다.
- 실패하면 화면을 초기화하지 않고 오류 live region을 표시하며 CTA 포커스를 유지한다.

### 4.2 첫 취향 선택

- 최소 3개, 최대 8개이며 스킵과 닫기가 없다.
- `3/8` 카운터가 유일한 진행 표기다. 의미 없는 점·막대 진행률을 추가하지 않는다.
- 칩은 `aria-pressed`를 갖는 다중선택 버튼이다.
- 선택값은 즉시 로컬 상태에 보존하고 제출 실패 시 유지한다.
- 완료하면 같은 `/`에서 카드 A 조준 상태로 전환한다.

### 4.3 카드 A/B 뽑기

- 초기에는 A만 활성화하고 A 공개 후 B를 활성화한다.
- 각 카드가 하나의 완성 아이디어다. 추가 조합·확정 단계는 없다.
- 중앙 덱은 클릭·포커스 대상이 아니며 비행 애니메이션의 시각적 출발점으로만 사용한다.
- 두 번째 카드 공개 후 `BF.D에게 골라달라고 하기`와 `다른 두 장`을 노출한다.
- 카드 안에 별도 스크롤을 만들지 않고 페이지가 스크롤을 소유한다.

### 4.4 BF.D 로그인·선택·칭찬

- 로그인 전 카드 내용은 읽을 수 있지만 선택 영역은 `inert`와 `aria-disabled` 상태다.
- 로그인 UI는 `role=dialog`, `aria-modal=true`, 레이블, 포커스 트랩, 복귀 포커스를 갖는다.
- 두 카드는 `radiogroup` 안의 `radio`이며 `aria-checked`와 좌우 방향키를 지원한다.
- 선택은 링과 체크를 주 신호로 쓰고, 미선택 카드도 읽을 수 있는 명도를 유지한다.
- 칭찬은 `나도 필요해`, `완성하면 알려줘`, `너라면 만들어` 중 하나만 선택한다.
- 중복 응원은 서버 idempotency key로 막는다.

## 7. 저장·오류 계약

- 서버 저장을 확인했을 때만 성공 체크를 표시한다.
- 오프라인이면 응원과 선택을 로컬 큐에 저장하고 `role=status`로 `응원을 기기에 저장했어요. 연결되면 자동으로 보낼게요.`를 알린다.
- 재시도 액션이 있는 스낵바는 사용자가 처리할 때까지 유지한다.
- 네이티브 공유 실패 시 링크 복사로 폴백하고, 복사까지 실패하면 URL을 직접 선택할 수 있게 노출한다.
- OAuth 복귀, 새로고침, 중복 탭 후에도 같은 라운드와 선택 상태를 복원한다.

## 8. K-factor 측정 계약

네이티브 공유 API는 실제 전달 인원수를 알려주지 않는다. 따라서 공유 버튼 클릭이나 `4명에게 전달`을 K의 입력으로 세지 않는다.

```text
K = origin당 고유 round_opened 수 x opened -> round_shared 전환율
목표 모델 = 3.2 unique opens x 31.185% = 0.998 ~= 1.00
```

`origin_round_id`는 직접 공유 출처 라운드 ID이며 세대마다 새 출처로 갱신한다. 전체 계보 분석이 필요하면 별도의 `root_round_id`를 보존한다.

- 최초 라운드는 `round_id = root_round_id`, `parent_round_id = null`이다.
- 수신자가 자기 카드 두 장을 완성하면 새 `child_round_id`를 만들고, 직전 공유 라운드를 `origin_round_id`와 `parent_round_id`로 사용한다.
- 다음 공유 링크에는 새 자식 라운드를 `round_id`로 넣고 최초 `root_round_id`는 그대로 보존한다.

| 이벤트 | 필수 속성 | K 포함 여부 |
|---|---|---|
| `share_sheet_opened` | `round_id`, `channel`, `occurred_at` | 제외, 진단용 |
| `round_opened` | `round_id`, `origin_round_id`, `opener_key`, `occurred_at` | 고유 오픈 분자 |
| `auth_completed` | `origin_round_id`, `user_id` | 퍼널 |
| `praise_sent` | `origin_round_id`, `user_id`, `candidate_id`, `praise_id` | 퍼널 |
| `own_flow_started` | `origin_round_id`, `user_id`, `seed_id` | 퍼널 |
| `two_cards_completed` | `origin_round_id`, `user_id`, `child_round_id` | 퍼널 |
| `round_shared` | `origin_round_id`, `user_id`, `child_round_id` | K 결과 |

`opener_key`는 로그인 전 익명 기기 키와 로그인 후 `user_id`를 조정해 같은 사람의 반복 오픈을 중복 집계하지 않는다. 운영 목표는 변동 여유를 둔 K>=1.1이다.

## 9. 구현 순서

1. 라우트별 상태 머신과 분석 이벤트 스키마를 먼저 고정한다.
2. 로그인 복귀와 첫 취향 전체 화면 상태를 구현한다.
3. 현재 덱 물성을 재사용해 A/B 직접 뽑기를 구현한다.
4. `/vs/[slug]` 로그인, 라디오 선택, 칭찬, 오프라인 큐를 구현한다.
5. 공유 링크에 `origin_round_id`를 연결하고 K 퍼널을 검증한다.
6. 모바일 브라우저, 키보드, reduced motion, 네트워크 실패를 회귀 테스트한다.

## 10. 완료 조건

- 320px, 360px, 390px 모바일 폭과 데스크톱에서 겹침·가로 잘림이 없다.
- 첫 로그인 후 취향을 3개 고르기 전에는 게임판으로 갈 수 없다.
- 카드 A와 B를 카드 표면 탭 두 번으로 완성하고 중앙 덱은 입력 대상이 아니다.
- 카드 비행·플립과 reduced-motion 대체 상태가 모두 동작한다.
- BF.D 카드는 마우스, 터치, Tab, 방향키로 한 장만 선택할 수 있다.
- 모든 주요 터치 타깃은 최소 48x48px다.
- 네트워크 실패를 성공처럼 표시하지 않고 로컬 큐 상태를 읽어준다.
- `share_sheet_opened`만으로 K가 증가하지 않으며 고유 오픈과 자식 `round_shared`로 K가 계산된다.
- 별도 확정·발행·대시보드 화면이 핵심 플로우에 다시 생기지 않는다.

## 11. 2026-07-10 구현·검증 기록

- `/`는 `MobileFlow`가 로그인, 첫 취향, 카드 A/B 게임판을 같은 라우트 상태로 관리한다.
- `/vs/[slug]`는 로그인 전 읽기 전용, 로그인 후 라디오 선택·칭찬·자기 카드 재진입을 제공한다.
- 라운드별 카드·칭찬 초안을 로컬에 저장해 새로고침과 중복 탭에서 복원한다.
- `duel_votes` 오프라인 큐는 idempotency key로 재전송하며, 실제 `project-bfd` 마이그레이션과 서버 저장을 검증했다.
- 인앱 브라우저에서 320/360/390px와 1280px, 포커스 복귀, 방향키, reduced motion, 네트워크 실패·복구, 콘솔 오류 0을 확인했다.
- `npm run build`, `npm run typecheck`, `npm run lint`가 모두 통과했다.

운영 전 잔여 조건은 Google provider 실 OAuth 왕복, 인증 기반 RLS 강화, BF.D 코호트 K>=1.1 실측이다.
