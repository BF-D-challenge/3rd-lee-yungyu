## 처음 보는 사람을 위한 한 문장

**맛핀**은 YouTube·Instagram·TikTok에서 발견한 맛집 영상 URL을 넣으면 식당 위치·대표 메뉴·원본 영상을 개인 지도에 정리하고, 오늘 갈 곳과 여행 동선을 고르게 해주는 모바일 퍼스트 웹 앱이다.

공개 영상을 보는 권리에 돈을 받지 않는다. 사용자가 돈을 내는 대상은 **흩어진 링크를 대신 정리하는 자동화, 반복 저장의 한도 해제, 저장한 장소로 여행 동선을 만드는 도구**다.

## 0-A. 포지셔닝 — UVP·타겟·차별점·플로우

> 작성 방법: ① 로컬 리서치의 Beli 5,800만 평가·영상→지도 경쟁군 5개+·국내 B2C 맛집 앱 직접 과금 성공 선례 0을 근거로 삼음 → ② “당신”의 저장함→지도 재검색 장면을 지금까지/이제부터 대조로 압축 → ③ 단일 디자인 시스템에 지도·동선 히어로 순간만 컨셉추얼하게 표현.

### UVP (통증 → 대조 훅)

> **당신은 오늘 갈 맛집을 고르려다 릴스 저장함, 유튜브 재생목록, 네이버지도를 몇 번 오갔나요?**  
> **지금까지:** 영상을 저장하고도 식당명과 주소를 다시 찾아 지도에 두 번째로 적었습니다.  
> **이제부터:** 영상 URL 1개를 넣고 식당·메뉴·원본 영상이 붙은 핀 1개를 확인한 뒤, 핀 2~6개로 오늘의 동선을 고릅니다.  
> **세 영상 플랫폼에서 당신이 고른 맛집 영상을 국내 지도 핀과 여행 동선으로 이어주는 유일한 개인 맛집 지도.** 오늘 영상 URL 하나로 첫 핀을 만듭니다.

### 타겟

**주 2~3회 YouTube·Instagram·TikTok에서 맛집 영상을 저장하고, 주말 약속·국내 여행 직전에 다시 찾아 지도 앱에 재입력하는 20~30대 모바일 사용자.** Beli의 5,800만 평가는 개인 맛집 리스트 수요를 보여주지만, 국내 B2C 맛집 앱의 직접 과금 성공 선례는 확인되지 않았으므로 초기 타겟은 “맛집 발견”이 아니라 **반복되는 저장 노동과 여행 직전의 결정**을 모두 겪는 사람으로 좁힌다.

### 기존에 잘되는 앱 vs 차별점

| 기존에 잘되는 앱 | 잘하는 것 | 맛핀이 남긴 빈틈 |
|---|---|---|
| **Beli** (2025 Q2 누적 평가 5,800만 건) | 방문한 식당의 랭킹·친구 피드·가고 싶은 목록을 입소문으로 키웠다. | 방문 **후** 기록이 코어라, 세 플랫폼의 영상을 국내 지도의 방문 후보로 바꾸는 입력 자동화는 없다. |
| **네이버지도·Google Maps** | 장소 저장·분류·공유와 길찾기를 무료로 잘한다. | 영상 저장함에서 식당명·주소를 다시 찾아 적는 **입력 노동**이 남는다. |
| **GeoTok·Stasht·MunchMap 등 5개+** | 영상→지도 전환과 OS 공유 시트 진입을 이미 보여줬고, 무료 진입이 표준이다. | 맛핀은 한국어 상호·국내 POI 확인과 저장한 핀 2~6개의 **여행 동선**에 범위를 좁힌다. |

유일 주장 하나로 압축: **“세 영상 플랫폼에서 당신이 고른 맛집 영상을 국내 지도 핀과 여행 동선으로 이어주는 유일한 개인 맛집 지도.”**

### 전체 플로우 (5단계)

**① 맛집 영상 URL 1개 붙여넣기 → ② 식당·주소·대표 메뉴 후보 확인 → ③ 원본 영상이 붙은 개인 지도 핀 저장 → ④ 핀 2~6개를 골라 오늘의 장소·이동 순서 결정 → ⑤ 읽기 전용 동선 공유·복사 또는 7일 여행 패스 선택**

## 1. 문제 정의 — Phase 2/7

### 표면 문제

맛집 영상을 여러 플랫폼에 저장해도 실제로 외식하거나 여행할 때 어느 영상에 어느 식당이 있었는지 다시 찾기 어렵다.

### 본질적 문제

영상 플랫폼의 저장 기능은 **다시 시청하기**에는 적합하지만 **현재 위치에서 갈 식당을 결정하기**에는 적합하지 않다.

1. YouTube·Instagram·TikTok의 저장함이 서로 분리돼 있다.
2. 영상 제목과 썸네일만으로는 식당 위치와 대표 메뉴를 비교할 수 없다.
3. 사용자는 기억나는 키워드로 영상을 다시 찾은 뒤 네이버·카카오 지도에 식당을 재검색한다.
4. 이중 입력이 귀찮아 대부분의 저장 영상은 방문 결정으로 이어지지 않는다.

> **본질 문제 한 문장:** 맛집을 발견한 순간과 실제로 방문할 곳을 고르는 순간 사이에, 영상 링크를 위치 기반 후보로 바꿔주는 개인 정리 도구가 없다.

### 현재 사람들이 푸는 방식

- 각 플랫폼의 저장·컬렉션·좋아요에 영상을 쌓아둔다.
- 카카오톡 나에게 보내기나 메모 앱에 URL을 모은다.
- 영상 설명·댓글에서 상호를 찾고 네이버지도·카카오맵에 다시 저장한다.
- 여행 직전에 저장 영상을 다시 훑거나, 결국 지도 검색 결과와 리뷰 앱만 본다.
- 튜브맵·맛튜버맵·유튜브 플레이스 같은 공개 맛집 지도를 사용한다. 다만 이는 사용자가 세 플랫폼에서 직접 고른 영상을 한 개인 지도에 합치는 도구와는 다르다.

### 왜 지금인가

- 모바일 공유 URL은 플랫폼이 달라도 사용자가 직접 전달할 수 있는 공통 입력이다.
- 언어 모델과 지오코딩을 조합하면 상호·지역·메뉴 후보를 제안해 수동 입력을 줄일 수 있다.
- 공개 영상과 공개 맛집 지도는 이미 무료 대안이 많다. 따라서 단순 열람이 아니라 **개인의 저장 노동을 없애는 것**으로 가치를 재정의해야 한다.

위 세 문장은 시장 성공의 증거가 아니라 지금 검증할 수 있게 된 조건이다. 수요와 결제 의향은 1주차 실험 전까지 가설로 취급한다.

## 2. 타겟과 사용자가 원하는 일 — Phase 3/7

### 핵심 사용자

**민지, 29세 직장인**

- 주 2~3회 YouTube·Instagram·TikTok에서 맛집 영상을 저장한다.
- 주말 약속이나 국내 여행을 준비할 때 저장한 영상부터 다시 찾는다.
- 네이버지도나 카카오맵에도 장소를 저장하지만 영상 저장함과 지도 저장 목록이 맞지 않는다.
- 친구에게 맛집 후보를 보낼 때 영상 링크 여러 개와 지도 링크를 따로 보낸다.
- 공개 인기순보다 자신이 믿는 크리에이터와 직접 저장한 영상을 우선하고 싶다.

### 제외하는 초기 사용자

- 맛집 영상을 거의 저장하지 않는 사람
- 공개 랭킹만 보고 식당을 정하는 사람
- 식당 예약·배달·리뷰 작성을 주목적으로 하는 사람
- 채널 전체를 자동으로 지도화하려는 유튜버

### 사용자가 원하는 일

> 여러 플랫폼에서 맛집 영상을 발견했을 때, URL 하나만 보내 식당과 메뉴를 내 지도에 남기고 싶다. 그래야 외식이나 여행을 계획할 때 저장 영상을 다시 뒤지지 않고 근처 후보를 비교해 오늘 갈 곳을 정할 수 있다.

### Job Story

When **주말 약속이나 여행지에서 먹을 곳을 정해야 할 때**,  
I want to **내가 저장한 맛집 영상을 위치와 메뉴 기준으로 한 번에 보고**,  
so I can **다시 검색하지 않고 믿을 만한 후보 하나와 이동 순서를 결정할 수 있다**.

### Value Proposition

- **Who**: 두 개 이상의 영상 플랫폼에 맛집 영상을 반복 저장하는 모바일 사용자
- **Why**: 저장은 많이 하지만 실제 방문 시점에는 영상을 찾고 식당을 재검색하는 수고가 발생한다.
- **What before**: 플랫폼 저장함, 나에게 보내기, 지도 앱을 오가며 같은 장소를 두 번 정리한다.
- **How**: URL에서 상호·위치·메뉴 후보를 만들고, 사용자가 한 번 확인하면 개인 지도와 여행 후보에 넣는다.
- **What after**: 현재 위치나 여행 지역을 열어 저장한 원본 영상과 식당을 비교하고 한 곳을 결정한다.
- **Alternatives**: 공개 유튜버 맛집 지도는 넓은 데이터 탐색에 강하고, 일반 지도 앱은 장소 관리에 강하다. 맛핀은 **내가 고른 세 플랫폼의 원본 영상과 장소를 한 번의 저장으로 연결하는 것**에 집중한다.

## 3. 임팩트 가설 — Phase 4/7

### 북극성 지표

**주간 방문 결정 완료 사용자 수**

다음 세 행동을 7일 안에 모두 한 고유 사용자 수로 정의한다.

1. 서로 다른 영상 URL을 3개 이상 저장한다.
2. 위치가 확인된 핀 3개 이상을 개인 지도에서 본다.
3. 핀 하나에서 **오늘 갈 곳으로 선택**한 뒤 원본 영상 또는 외부 길찾기를 연다.

단순 가입·지도 조회·영상 재생은 완료로 세지 않는다.

### 4주 검증 목표

| 지표 | 목표 | 의미 |
|---|---:|---|
| 초대 테스트 사용자 | 30명 | 작은 행동 코호트 확보 |
| 활성화율 | 60% 이상 | 첫 10분 안에 위치가 확인된 핀 3개 저장 |
| 자동 추출 성공률 | 80% 이상 | 상호·지역 후보를 제시하고 한 번의 확인으로 저장 |
| URL 1개 확인 시간 | 중앙값 45초 이하 | 직접 지도 검색보다 가볍게 느껴질 기준 |
| 7일 내 방문 결정 완료율 | 활성 사용자의 35% 이상 | 저장이 실제 결정으로 이어지는지 확인 |
| D7 재방문율 | 25% 이상 | 월 구독 가설을 유지할 최소 신호 |
| 유료 화면 진입률 | 활성 사용자의 10% 이상 | 자동정리 한도·여행 동선에 관심이 있는지 확인 |
| 실제 결제율 | 활성 사용자의 3% 이상, 최소 1명 | 말이 아닌 결제 행동 확인 |

### 정성적 임팩트

> “저장해둔 영상이 어디 있었는지 찾지 않아도, 여행지에서 바로 후보를 고를 수 있어요.”

### 성장 루프

1. 사용자가 영상 링크를 저장해 개인 핀을 만든다.
2. 여행 후보 2~6개를 골라 읽기 전용 동선 카드로 공유한다.
3. 동행자가 카드를 열어 장소를 자기 맛핀에 복사한다.
4. 수신자가 자기 영상 링크를 추가하고 새 동선을 만든다.
5. 새 동선이 다시 공유된다.

공유는 개인 지도를 공개하는 행위가 아니다. 사용자가 고른 핀과 동선만 별도 링크로 공개한다.

### K 측정 원칙

```text
K_content = 활성 사용자 1명당 발송한 동선 링크 수
            × 수신자 오픈율
            × 수신자의 맛핀 활성화율
```

- 4주 MVP에서는 `K_content`를 계측만 한다.
- 실제 활성 사용자 100명 이상인 두 개의 연속 코호트에서 `K_content ≥ 1`이 확인되기 전까지 “바이럴 K가 높은 앱”이라고 표현하지 않는다.
- 1주차에 공유가 발생하지 않아도 핵심 가치 전체를 폐기하지는 않는다. 대신 성장 채널을 제품 바이럴이 아닌 콘텐츠·검색·커뮤니티 유입으로 다시 설계한다.

## 4. 솔루션과 MVP — Phase 5/7

### 솔루션 형태

설치 없이 링크로 바로 진입할 수 있는 **모바일 퍼스트 웹 앱**으로 만든다. 첫 저장은 게스트 상태에서 허용하고, 두 번째 저장 또는 동기화 시 이메일 매직링크로 계정을 보존한다.

네이티브 공유 확장보다 붙여넣기 흐름을 먼저 검증한다. OS 공유 시트 연동과 네이티브 앱은 MVP 이후다.

### 핵심 가치 한 문장

**여러 영상 플랫폼에 맛집을 저장하는 사람을 위한 개인 지도인데, URL 하나를 넣으면 위치·메뉴·원본 영상이 함께 정리되고 저장한 장소로 여행 동선까지 만든다.**

### Must-have 기능 — 정확히 3개

#### 1. 영상 URL 저장과 장소 후보 추출

- 사용자가 제출한 YouTube·Instagram·TikTok URL만 처리한다.
- 원본 URL, 플랫폼, 크리에이터, 썸네일·제목 등 허용된 메타데이터를 보존한다.
- 영상과 공개 메타데이터에서 식당명·지역·대표 메뉴 후보를 만든다.
- 후보의 확신도를 보여주고 사용자가 저장 전에 상호·주소·메뉴를 확인한다.
- 추출에 실패하면 `식당명 / 지역 또는 주소 / 대표 메뉴` 세 칸을 직접 입력해 흐름을 끝낼 수 있다.
- 같은 식당에 여러 영상이 저장되면 새 핀을 만들지 않고 원본 영상 목록을 묶는다.

#### 2. 개인 맛집 지도와 방문 결정

- 위치가 확인된 저장 항목을 비공개 개인 지도에 핀으로 표시한다.
- 지도 줌에 따른 동적 클러스터와 현재 위치 기준 1km·3km·5km 반경을 제공한다. 5km 고정 격자로 데이터를 나누지는 않는다.
- 플랫폼, 음식 종류, `가고 싶음 / 다녀옴` 상태로 필터링한다.
- 핀 상세에서 대표 메뉴, 저장한 이유, 원본 영상 목록을 보고 `오늘 갈 곳으로 선택`, 외부 길찾기, 원본 영상 열기를 실행한다.
- 사용자의 명시적 공유 전에는 저장 URL과 개인 메모를 공개하지 않는다.

#### 3. 여행 후보 선택과 동선 만들기

- 저장한 핀 2~6개와 출발 지역을 골라 방문 순서 초안을 만든다.
- MVP의 동선은 거리 기반 단순 정렬이며 실시간 교통·영업시간을 보장하는 내비게이션이 아니다.
- 최종 길찾기는 외부 지도 앱으로 넘긴다.
- 사용자가 고른 핀만 읽기 전용 동선 카드로 공유하고, 수신자는 장소 하나 또는 전체를 자기 맛핀으로 복사할 수 있다.
- 무료 사용자는 결과 미리보기를 볼 수 있고, 전체 순서 저장과 반복 생성은 유료 가설로 둔다.

### 최소 데이터 구조

| 객체 | 핵심 필드 |
|---|---|
| `saved_video` | 사용자, 원본 URL, 플랫폼, 제목, 크리에이터, 썸네일, 저장 시각 |
| `place_candidate` | 식당명, 주소, 위도·경도, 대표 메뉴, 추출 확신도, 사용자 확인 여부 |
| `restaurant_pin` | 사용자, 장소 식별자, 상태, 개인 메모, 연결된 원본 영상 목록 |
| `trip_route` | 사용자, 제목, 선택 핀, 방문 순서, 공유 상태, 공개 토큰 |
| `event` | import_started, import_confirmed, pin_viewed, decision_completed, paywall_viewed, route_shared, pin_copied |

### Demo Flow

1. 사용자가 Instagram 맛집 영상 URL을 붙여 넣는다.
2. 맛핀이 식당명·지역·메뉴 후보를 보여주고 사용자가 한 번 확인한다.
3. 같은 방식으로 YouTube와 TikTok 링크를 저장하자 개인 지도에 핀이 생긴다.
4. 현재 위치 5km 안의 핀을 열어 원본 영상과 메뉴를 비교하고 한 곳을 오늘 갈 곳으로 선택한다.
5. 여행 후보 세 곳의 방문 순서를 만들고 친구에게 공유한다. 친구는 장소를 자기 지도에 복사한다.

### Non-goals

- 유튜버 채널·플랫폼 전체를 크롤링하거나 사용자가 요청하지 않은 영상을 대량 수집하지 않는다.
- 영상 파일을 다운로드·재호스팅하거나 원본 플랫폼을 대체하는 플레이어를 만들지 않는다.
- 공개 범용 맛집 DB, 별점·리뷰 랭킹, 예약·웨이팅·배달 기능을 만들지 않는다.
- 유튜버용 자동 지도 SaaS, 식당용 대시보드, 광고·스폰서 노출은 후순위이며 이번 MVP에서 만들지 않는다.
- 네이티브 iOS·Android 앱, OS 공유 확장, 오프라인 지도는 만들지 않는다.
- 실시간 교통·영업시간까지 반영한 최적 경로를 약속하지 않는다.
- 친구와 동시에 편집하는 공동 지도와 공개 소셜 피드는 만들지 않는다.

### 제품 원칙

1. **개인 지도는 비공개가 기본이다.** 공유 범위는 사용자가 직접 선택한다.
2. **원본을 지운 채 정보만 가져오지 않는다.** 모든 핀에서 원본 플랫폼과 영상을 다시 열 수 있다.
3. **틀린 자동화보다 빠른 확인이 낫다.** 주소가 모호하면 임의 저장하지 않고 후보를 고르게 한다.
4. **첫 가치는 로그인 전에 보여준다.** 첫 URL의 장소 후보를 본 뒤 계정 보존을 요청한다.
5. **공개 데이터 열람을 잠그지 않는다.** 결제는 사용자의 정리량과 여행 작업을 줄이는 데 연결한다.

### B2C 수익 가설

#### 무료

- 월 10개 자동 추출
- 직접 입력한 핀과 개인 지도 열람
- 원본 영상 열기와 오늘 갈 곳 선택
- 여행 동선 한 번 미리보기

#### 여행 패스 — 7일 3,900원 가설

- 7일 동안 자동 추출 50개
- 여행 동선 3개 저장·재생성
- 읽기 전용 동선 공유와 전체 장소 복사

여행 준비가 간헐적이라면 월 구독보다 한 번의 필요 순간에 결제받는 모델이 더 현실적이라는 가설이다.

#### 맛핀 Plus — 월 4,900원 가설

- 정상적인 개인 사용 범위에서 자동 추출 무제한
- 여행 동선 무제한
- 고급 필터와 내보내기

Plus는 D7·D30 반복 사용이 확인된 뒤 유지한다. D7 재방문이 25% 미만이면 월 구독을 주력으로 두지 않는다.

#### 수익성 가드레일

- 공개 영상 열람·기본 지도에 과금하지 않는다.
- 무료 자동 추출량으로 변동 비용을 통제한다.
- 결제 사용자 1명당 추출·지도·AI 변동 비용은 순매출의 20% 이하를 목표로 한다.
- “무제한”은 정상적인 개인 사용을 뜻한다. 자동화 공격과 대량 상업 수집은 속도 제한·공정 사용 정책을 적용한다.
- 4주 MVP에서는 두 가격의 실제 사전 결제를 검증하되, 정식 자동결제·환불 시스템은 결제 신호 확인 뒤 구현한다.

### 가장 위험한 가정

1. **사용자가 실제로 URL을 가져와 넣을 만큼 문제가 절실한가?** 인터뷰에서 불편하다고 말하는 것과 링크 5개를 제출하는 것은 다르다.
2. **자동정리와 여행 동선에 실제로 돈을 내는가?** 공개 영상과 일반 지도는 무료이므로 편리함만으로는 약할 수 있다.
3. **Instagram·TikTok까지 포함했을 때 추출 정확도와 정책 준수가 가능한가?** 기술적으로 불안정하면 핵심 경험이 수동 입력으로 퇴행한다.

### 1주차 무코드 검증 실험

#### 모집 조건

- 최근 3개월 동안 맛집 영상 10개 이상을 저장한 사람 20명
- 두 개 이상의 플랫폼을 사용하는 사람을 우선 모집
- 인터뷰 의향이 아니라 실제 저장 URL을 제출할 수 있는 사람만 포함

#### 실험 절차

1. 참가자가 자기 저장함에서 영상 URL 5개를 폼으로 제출한다.
2. 운영자가 코드를 쓰지 않고 상호·위치·메뉴·원본 링크를 정리한 개인 지도를 만들어 돌려준다.
3. 참가자는 기존 저장 방식과 개인 지도를 각각 사용해 지정 지역에서 갈 곳 하나를 고른다. 결정 시간과 재검색 횟수를 잰다.
4. 참가자는 마음에 드는 2~6곳으로 여행 동선 카드를 만들고 동행자 한 명에게 보낸다.
5. `7일 여행 패스 3,900원`과 `Plus 월 4,900원`을 정확히 표시하고, 미출시 제품임을 밝힌 환불 가능한 사전 결제를 받는다.
6. 7일 뒤 지도 재방문, 새 URL 추가 의향이 아닌 실제 추가 행동을 확인한다.

#### 통과 기준

| 가정 | 통과 기준 |
|---|---:|
| 문제 빈도 | 20명 중 15명 이상이 요청 후 24시간 안에 URL 5개 제출 |
| 개인 지도 가치 | 지도 사용자의 70% 이상이 3분 안에 한 곳 결정 |
| 대안 대비 개선 | 기존 방식 대비 결정 시간 중앙값 30% 이상 감소 |
| 위치화 가능성 | 제출 URL의 80% 이상에서 2분 안에 식당과 지점 후보 식별 |
| 결제 의향 | 20명 중 2명 이상이 두 유료안 중 하나를 실제 사전 결제 |
| 반복성 | 7일 안에 25% 이상이 새 URL을 실제로 추가 |
| 성장 루프 프록시 | 4명 이상이 동선 공유, 수신자 2명 이상이 장소 복사, 1명 이상이 자기 URL 저장 |

|---|---|
| 20명 중 10명 미만이 URL 5개를 제출 | 광범위한 B2C 문제 가설 폐기. 여행 직전 사용자처럼 더 좁은 타겟이 없다면 프로젝트 중단 |
| 지도 사용자의 50% 미만이 3분 안에 한 곳을 결정하거나 결정 시간이 20%도 줄지 않음 | 지도 기반 방문 결정 가설 중단. 단순 저장함으로 축소하지 말고 문제를 다시 정의 |
| 두 가격을 본 적격 사용자 20명 중 실제 사전 결제 0명 | 유료 B2C 모델 중단. 무료 제품의 제휴·예약 수익을 별도 검증하거나 프로젝트 중단 |
| 특정 플랫폼이 제출 URL의 70% 이상이고 다른 플랫폼의 위치화 성공률이 60% 미만 | 첫 버전을 해당 플랫폼 하나로 축소하고 나머지는 직접 입력으로 전환 |
| 전체 URL 위치화 성공률이 70% 미만 | 세 플랫폼 자동 추출 약속 중단. YouTube 우선 또는 사용자 확인 중심 제품으로 피벗 |
| D7 새 URL 추가율이 20% 미만이지만 여행 패스 결제가 있음 | 월 구독 폐기, 7일 패스와 여행 단위 결제에 집중 |
| 공유·복사 행동이 0건 | high-K 포지셔닝 폐기. 핵심 사용·결제가 통과하면 제품은 유지하고 외부 획득 채널 설계 |
| 사용자당 변동 비용이 순매출의 30%를 넘음 | 무료 한도·추출 파이프라인·가격을 고치기 전 확장 중단 |

### 정책·데이터 안전 가드레일

- 사용자가 명시적으로 제출한 URL만 처리한다.
- 크리에이터 프로필·채널을 자동 순회하거나 로그인을 우회해 수집하지 않는다.
- 영상 파일을 저장하지 않고 원본 URL·허용된 임베드·메타데이터만 사용한다.
- 플랫폼별 공식 API·임베드·약관이 허용하는 범위를 먼저 사용한다.
- 자동 추출 근거가 부족하면 사용자가 직접 확인하게 하고, 틀린 주소를 확정값처럼 노출하지 않는다.
- 사용자는 저장 링크·개인 메모·여행 카드와 계정을 삭제할 수 있어야 한다.

## 6. 주차별 마일스톤 — Phase 7/7

### 1주차: 코드 없이 가장 위험한 가정 검증

- **목표**: URL 제출 행동, 방문 결정 개선, 두 유료안의 실제 사전 결제를 확인한다.
- **산출물**:
  - 적격 사용자 20명 목록과 동의 기록
  - 사용자별 수동 개인 지도 20개 또는 제출 완료자 전원의 지도
  - 기존 방식 대비 결정 시간 표
  - 사전 결제·공유·D7 행동 결과표
  - `진행 / 타겟 축소 / 수익 모델 피벗 / 중단` 결정문
- **게이트**: 결제 0명 또는 방문 결정 가치가 Kill 기준이면 2주차 개발에 들어가지 않는다.

### 2주차: URL 입력에서 확인된 핀까지

- **목표**: 사용자 제출 URL 한 개가 확인 가능한 장소 후보가 되어 DB에 저장되는 세로 한 줄을 만든다.
- **구현 범위**:
  - 게스트 세션과 이메일 매직링크
  - 플랫폼 URL 판별
  - 메타데이터·장소 후보 추출 인터페이스
  - 실패 시 세 필드 직접 입력
  - 저장·수정·삭제
- **산출물**: 세 플랫폼 각 10개 샘플의 성공률·수정 횟수 리포트와 모바일 데모 URL

### 3주차: 개인 지도와 방문 결정

- **목표**: 저장한 핀을 위치 기준으로 비교하고 한 곳을 실제 후보로 결정한다.
- **구현 범위**:
  - 동적 클러스터와 1km·3km·5km 반경
  - 플랫폼·음식·방문 상태 필터
  - 같은 장소의 영상 묶기
  - 핀 상세, 원본 열기, 외부 길찾기, 오늘 갈 곳 선택
  - 핵심 이벤트 계측
- **산출물**: 사용자 10명이 각 3개 핀을 저장하고 결정 완료까지 수행한 세션 기록

### 4주차: 여행 동선·유료 가설·공유 루프

- **목표**: 저장 → 결정 → 동선 → 공유 → 복사의 한 바퀴를 실제 사용자로 검증한다.
- **구현 범위**:
  - 핀 2~6개 거리 기반 순서
  - 여행 패스·Plus 유료 화면과 이벤트 계측
  - 읽기 전용 동선 카드
  - 수신자의 핀 복사와 첫 URL 저장 CTA
  - 오류·빈 상태·개인정보 삭제
- **산출물**:
  - 배포 URL
  - 3~5분 데모 영상
  - 30명 행동 퍼널
  - 수익 가설·K·D7 결과와 다음 버전 결정문

## 7. 최종 검수 체크리스트

### 기능

- [ ] YouTube·Instagram·TikTok 사용자 제출 URL을 구분한다.
- [ ] 자동 추출이 실패해도 세 필드 입력으로 핀을 저장할 수 있다.
- [ ] 같은 식당의 여러 원본 영상이 한 핀에 묶인다.
- [ ] 5km 반경과 지도 영역에서 저장 핀을 찾을 수 있다.
- [ ] 오늘 갈 곳 선택과 외부 길찾기가 계측된다.
- [ ] 여행 동선 카드에는 사용자가 고른 정보만 공개된다.
- [ ] 수신자가 핀을 복사하고 자기 URL을 추가할 수 있다.

### 수익·성장

- [ ] 무료 지도와 원본 영상 열람을 결제로 막지 않는다.
- [ ] 여행 패스와 Plus의 가격·혜택이 다르다.
- [ ] paywall_viewed와 purchase_started를 구분해 기록한다.
- [ ] 공유 수, 수신자 오픈, 수신자 활성화로 K를 계산한다.
- [ ] 데이터가 없는데 high-K·높은 결제율을 주장하지 않는다.

### 신뢰·정책

- [ ] 모든 장소에 원본 URL과 플랫폼 출처가 남는다.
- [ ] 확신도가 낮은 위치는 사용자 확인 없이 핀으로 확정하지 않는다.
- [ ] 영상 다운로드·재호스팅·채널 일괄 크롤링을 하지 않는다.
- [ ] 개인 지도는 기본 비공개다.
- [ ] 계정과 저장 데이터를 삭제할 수 있다.
