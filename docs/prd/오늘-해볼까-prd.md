## 처음 보는 사람을 위한 한 문장

**오늘 해볼까**는 검증된 해외 제품의 작동 방식을 네 장의 카드로 뽑아 **80~99%는 남기고 한 가지만 바꾼 아이디어**를 만들고, 바로 제작할 문구를 받은 뒤 지인에게 공유해 **매일 익명의 칭찬 카드 한 장**을 여는 모바일 퍼스트 웹 앱이다. 제품 메커니즘이 해외 원본과 완전히 같아도 허용하지만, 모든 결과는 1인이 바이브 코딩으로 반나절~2일 안에 배포 가능한 MVP로 줄인다.

사용자는 어려운 취향 조사를 하지 않는다. 먼저 뽑고, 마음에 들 때까지 전체 또는 한 장만 다시 뽑는다.

### UVP (통증 → 대조 훅)

> **ChatGPT에 "아이디어 좀"을 몇 번이나 물어봤나요? 그래서 몇 개를 만들었나요?**
> 질문 100번에 0개 — 오늘 해볼까는 **카드 4장에 오늘 만들 1개**입니다.
> 이미 돈이 벌리는 해외 제품에서 80~99%는 남기고 **한 끗만** 바꿉니다. 같아도 됩니다. 대신 혼자 만들 수 있는 가장 작은 작동 버전부터 시작합니다.

### 타겟

**AI 코딩 도구는 있는데 뭘 만들지 못 정해서 시작을 못 한 1인 빌더·사이드프로젝트 지망생 (20~30대).** 통증 유형은 개인적(아이디어 목록만 쌓이는 정체감) — 이미 ChatGPT에 묻는 행동이 있으므로 교육이 필요 없는 세그먼트다.

### 기존에 잘되는 것 vs 차별점

| 잘되는 것 | 잘하는 것 | 빈틈 |
|---|---|---|
| **ChatGPT** (기본값) | 무한한 아이디어 답변 | 물을수록 평범·거대해지고, 실행 문구도 응원도 없음 |
| **TrustMRR/acquire** (실매물 1,863건) | "실제 돈 내는 문제"의 증거 | 목록일 뿐, 내가 만들 것으로 변환해주지 않음 |
| **Gas** (익명 칭찬, 미국 고교 폭발 성장) | 익명 칭찬의 수신 기대감 | 칭찬이 실행과 연결 안 됨 — 리텐션 붕괴로 종료 |

유일 주장 하나로 압축: **"검증된 제품에서 한 끗만 바꿔 오늘 만들 것을 정해주고, 지인의 익명 칭찬으로 계속 만들게 하는 유일한 도구."** (아이디어 생성기라고 말하지 않는다 — 생성이 아니라 출발+지속이 상품이다.)

**네 장 뽑기(검증된 원본·돈 낼 사람·순간·한 끗) → 결과 이해 + 제작 문구 앞부분 → 공유하면 전체 문구 열림 → 지인의 익명 칭찬 도착 → 매일 한 장씩 뒤집으며 재방문**

## 1. 문제 정의

### 겉으로 보이는 문제

AI 코딩 도구로 제품을 만들기는 쉬워졌지만, 1인 빌더는 무엇을 만들지 정하지 못해 시작하지 못한다.

### 더 깊은 문제

완전히 새로운 아이디어를 만들려 하기 때문에 선택지가 끝없이 늘어난다. ChatGPT에 질문할수록 평범하거나 범위가 큰 답이 나오고, 사용자는 다시 질문만 반복한다.

실제로 필요한 것은 새로운 발명보다 다음 세 가지다.

1. 이미 사람들이 쓰고 돈을 내는 제품의 핵심 흐름
2. 누구에게 언제 필요한지 보여주는 구체적인 맥락
3. 원본을 망가뜨리지 않는 한 가지 변화

### 현재 해결 방식

- ChatGPT나 Claude에 아이디어를 계속 다시 물어본다.
- 아이디어 목록만 쌓고 무엇을 만들지 고르지 못한다.
- 성공한 제품을 보지만 어떤 부분을 남기고 바꿔야 할지 모른다.
- 첫 제작 문구를 직접 작성하다 범위가 커진다.
- 지인에게 막연히 “어때?”라고 물어보고 답을 받지 못한다.

### 왜 지금인가

AI 코딩으로 제작 비용이 낮아지면서 경쟁력은 코딩 속도보다 **좋은 출발점과 실행을 계속하게 만드는 힘**으로 이동했다. 로컬 리서치에서도 순수 아이디어 생성 자체보다 검증된 제품 구조, 구체적인 실행 문구, 공유 후 반응이 더 강한 가치로 확인됐다.

> AI 코딩 도구로 무언가 만들고 싶지만 아이디어를 못 정한 순간, 이미 잘되는 제품에서 출발한 네 장을 재미있게 뽑고 한 가지만 바꾼 결과를 얻어, 오늘 바로 만들기 시작하고 싶다.

| 사용 전 | 사용 후 |

| 결과가 왜 좋은지 이해하지 못한다. | 원본·고객·필요 순간·변화를 색과 라벨로 구분한다. |
| “프롬프트”라는 잠긴 상자만 본다. | 실제 첫 문장을 보고 제작 결과를 예상한다. |
| 혼자 정하고 실행을 미룬다. | 링크를 공유하고 익명 응원을 받아 다시 들어온다. |

## 3. 제품 원칙

### 원칙 1. 완전히 새로 만들지 않는다

결과 아이디어는 검증된 해외 제품의 핵심 입력 → 처리 → 결과 흐름을 80~99% 유지한다. 제품 메커니즘이 완전히 같아도 허용하고, 달라진다면 한 가지뿐이다.

### 원칙 2. 설명보다 먼저 뽑게 한다

첫 진입에 취향 조사, 로그인, 고정 기능 설명을 두지 않는다. 네 장 뽑기가 첫 행동이다.

### 원칙 3. 어려운 이름 대신 풀어서 설명한다

Character.AI, 1Lookup 같은 원본 이름만 보여주지 않는다. 고등학생도 이해할 수 있도록 다음처럼 설명한다.

> “대화할 상대를 고르면 이전 대화를 기억하며 관계가 이어지는 서비스”

원본 이름은 근거 표기로 보조한다.

### 원칙 4. 아이디어 카드와 칭찬 카드의 역할을 섞지 않는다

- 네 장의 조합 카드는 아이디어를 만드는 도구다.
- 칭찬 카드는 아이디어를 공유한 뒤 받은 긍정 반응을 하루 한 장씩 보여주는 보상이다.
- 모닥불·유령·의자·불꽃은 실제 앱에서 사용하지 않고 과거 실험 레퍼런스로만 보존한다.

### 원칙 5. 모방을 전면에 내세우지 않는다

사용자에게는 “따라 만들기”보다 **검증된 구조에서 출발한다**고 설명한다. 원본을 그대로 복제하지 않고 대상·순간·입력·출력 중 하나를 바꾼다.

### 원칙 6. 모든 보상은 실제 행동과 연결한다

- 공유 완료 → 전체 제작 문구 열림
- 익명 칭찬 도착 → 카드 큐에 들어가고 배정된 날짜에 한 장 공개
- 이름 확인 결제 → 응원을 보낸 사람의 이름을 일찍 확인

가짜 알림이나 가짜 응원은 절대 만들지 않는다.

## 4. 네 장의 카드

### 확정 분류

| 카드 | 라벨 | 답하는 질문 | 예시 |
|---|---|---|---|
| 1 | **검증된 원본** | 이미 어떤 제품 흐름이 작동하는가? | “정보 하나를 넣으면 상대 정보를 찾아 정리해주는 웹 서비스” |
| 2 | **돈 낼 사람** | 이 문제를 반복해서 겪고 돈을 낼 사람은 누구인가? | “매주 잠재 고객을 찾는 1인 영업자” |
| 3 | **필요한 순간** | 해결 욕구가 가장 커지는 때는 언제인가? | “첫 미팅 10분 전” |
| 4 | **한 끗 변화** | 무엇 하나를 더하거나 빼거나 바꿀 것인가? | “긴 리포트를 빼고 30초 음성만 제공” |

카드 1도 고정하지 않는다. 웹·모바일 앱·데스크톱 앱·브라우저 플러그인·키보드 확장 등 서로 다른 원본이 계속 바뀐다.

### 조합 규칙

1. 카드 1의 핵심 입력 → 처리 → 결과는 보존한다.
2. 카드 2와 3은 실제 반복 행동이 있는 구체적인 사람과 순간이어야 한다.
3. 카드 4는 추가·제거·교체 중 하나만 허용한다.
4. 기능을 두 개 이상 새로 붙이지 않는다.
5. 원본과 완전히 같거나 원본 충실도가 80% 이하인 결과는 폐기한다.
6. 결과 형태는 웹·앱·플러그인 중 사용 순간에 가장 맞는 하나를 제안한다.
7. 근거가 모호한 원본은 카드 덱에 넣지 않는다.

### 데이터 원칙

- 기존 카드는 전량 폐기하고 다시 만든다.
- TrustMRR 감사 통과 제품, App Store 국가별 상위 앱, 상세 리서치가 있는 제품만 원본 후보로 쓴다.
- 제품 이름만 수집하지 않고 대상·문제·입력·처리·결과·과금 단위까지 기록한다.
- 카드 결과에는 “무엇을 그대로 남겼는지”와 “무엇 하나를 바꿨는지”를 함께 저장한다.
- 추천 런타임에는 감사 통과 카드만 노출한다.
- **순간·한 끗 카드도 감사한다: 축소 중인 채널·행동(예: 개인 블로그 유입 의존)에 앵커된 문구는 폐기하고, 지금 늘거나 유지되는 행동(연재 플랫폼·상품 페이지·숏폼·메신저)에만 앵커한다.** 원본이 검증돼도 순간·한 끗이 사양 행동이면 조합 전체가 죽는다.

## 5. 카드 뽑기 경험

### 반드시 지킬 동작

1. 부채꼴 카드 덱은 360도로 계속 회전한다.
2. 카드를 뽑으면 덱에서 결과 슬롯으로 날아간다.
3. 네 장을 한꺼번에 다시 뽑을 수 있다.
4. 각 카드만 따로 바꿀 수 있다.
5. 각 카드를 누르면 같은 분류의 후보가 아래에 펼쳐진다.
6. 사용자는 카드 대신 직접 문장을 입력할 수 있다.
7. 고정 기능과 취향 조사는 넣지 않는다.
8. 네 카드의 라벨과 색을 결과 문장에서도 그대로 사용한다.
9. 화면 아래 주요 버튼은 모바일 화면 높이 안에서 찾기 쉬워야 한다.
10. 데스크톱에서는 네 카드와 결과를 한눈에 비교할 수 있어야 한다.

### 결과 화면이 답해야 하는 것

- 원본 제품은 무엇을 해주는가?
- 누가 왜 돈을 내는가?
- 어떤 순간에 가장 필요할까?
- 무엇 하나를 바꿨는가?
- 결과는 웹·앱·플러그인 중 무엇인가?
- 오늘 만들 가장 작은 화면은 무엇인가?

### 결과 아이디어의 출력 구조 — 6블록 브리프

카드로 조합된 모든 결과 아이디어는 0-A 포지셔닝 섹션과 같은 구조의 **6블록 브리프**로 조립된다. 재료는 전부 네 카드 데이터에서 나오며, 블록별 매핑은 다음과 같다.

| 블록 | 내용 | 데이터 출처 |
|---|---|---|
| 1. UVP | 순간의 통증 + "검증된 해외 원본 80~99% 유지, 완전 동일 허용" 선언 | 순간 카드 + 원본·한 끗 카드 |
| 2. 타겟 | 누가, 어떤 반복 행동을 하는가 | 돈 낼 사람 카드 |
| 3. 해외에서 잘되는 원본 vs 이번 적용 | 원본이 잘하는 것(매출 근거 표기) → 이번 타겟·순간 → 명확한 핵심 주장. 기능 차별화는 필수 아님 | 원본 카드 evidence + 순간·한 끗 카드 |
| 4. 전체 플로우 | 보존 흐름 → 한 끗 → 오늘 만들 최소 화면 | 원본 preservedFlow + 한 끗 smallestBuild |
| 5. AI 코딩 도구 제작 지시 | 1인 반나절~2일 MVP 하드 게이트·상태 3종·가짜 데이터 금지 | 전 카드 조합 |
| 6. 플로우 이미지 프롬프트 | 단일 디자인 시스템 5패널 스토리보드 (히어로 1·5번만 컨셉추얼) | 보존 흐름 3단계 + 한 끗 + 최소 화면 |

제목·훅·블록 1(UVP)까지가 공유 전 미리보기이고, 블록 2(타겟)부터 6까지가 공유 후 열리는 잠금 영역이다.

### 제작 문구 공개 방식

잠긴 영역의 제목을 단순히 “프롬프트”라고 쓰지 않는다.

- 제목: **AI 코딩 도구에 붙여 넣을 제작 문구**
- 공개: 첫 2~3줄
- 잠금: 이후 내용을 가볍게 블러 처리
- 해제: 아이디어 링크 공유 완료
- 해제 후 행동: 전체 복사

공유 보상은 무료이며 사용자 획득 장치다. 이름 확인 결제와 같은 화면에 두지 않는다.

## 6. 매일 한 장 익명 칭찬 카드

### 핵심 의미

칭찬 카드는 “누군가 내 실행을 보고 남긴 긍정적인 한마디를 매일 한 번 기대하며 확인하는 보상”이다.

### 공개 규칙

- 실제로 도착한 칭찬만 카드 큐에 넣는다. 가짜 칭찬은 만들지 않는다.
- 현지 날짜 기준으로 매일 한 장만 무료로 공개한다.
- 오늘 카드는 뒷면에서 앞면으로 직접 뒤집어 본다.
- 다음 카드는 뒤에서 일부 보이지만 문구와 보낸 사람 정보는 노출하지 않는다.
- 오늘 카드가 없으면 결제 버튼 대신 **칭찬 받아오기** 공유를 권한다.

### Solitaire 이미지 문법

- 짙은 테이블 위에 오늘 카드와 잠긴 다음 카드를 겹쳐 놓는다.
- 카드 앞면과 뒷면의 대비, 카드 딜, 한 장 뒤집기, 다음 카드 일부 노출을 사용한다.
- 특정 Solitaire 앱의 카드 문양이나 아트를 복제하지 않는다.
- 수트·카드 테두리·뒷면 패턴은 오늘 해볼까의 독자적인 그래픽으로 만든다.

### 응원 내용과 이름

- 응원 내용은 긍정 문구만 허용한다.
- 받는 사람에게 발신자 이름은 즉시 보이지 않는다.
- 기본 표시는 **익명의 응원**이다.
- 발신자가 이름 공개에 동의한 경우에만 30일 뒤 무료로 공개한다.
- 바로 확인하려면 오늘 칭찬 카드에서 **누가 보냈는지 보기** 결제 의사를 묻는다.
- 영구 익명을 선택한 발신자는 결제해도 공개하지 않는다.
- 잠긴 다음 카드가 있을 때만 **다음 칭찬 미리 보기** 결제 의사를 별도로 묻는다.
- 두 버튼은 서로 다른 상품이며 MVP에서는 지불 의사를 측정하는 가짜 문으로만 운영한다.

## 7. 전체 흐름

### Flow A — 아이디어를 만들고 공유하는 사람

| 화면 | 목적 | 핵심 화면 변화 | 주요 행동 |
|---|---|---|---|
| A1. 네 장 뽑기 | 재미있게 출발 | 회전하는 부채꼴 덱에서 네 장이 슬롯으로 날아감 | 전체 다시 뽑기, 한 장 바꾸기, 직접 쓰기 |
| A2. 결과 이해 | 원본과 한 끗을 이해 | 네 색 라벨이 결과 문장과 연결되고 제작 문구 앞부분이 보임 | 이번에 할 작은 실행 고르기 |
| A3. 공유 완료 | 무료 보상 제공 | 공유가 끝나며 전체 제작 문구가 열림 | 전체 복사, 오늘의 칭찬 보기 |
| A4. 칭찬 카드 비어 있음 | 실제 칭찬 획득 | 빈 카드 스택과 칭찬 받아오기 안내 | 링크 다시 공유 |
| A5. 오늘의 칭찬 | 재방문과 기대감 | 오늘 카드 한 장을 뒤집고 다음 잠금 카드가 일부 보임 | 칭찬 읽기, 보낸 사람 확인, 다음 카드 미리 보기 |

### Flow B — 공유 링크를 받은 사람

| 화면 | 목적 | 핵심 화면 변화 | 주요 행동 |
|---|---|---|---|
| B1. 친구의 실행 보기 | 무엇을 칭찬하는지 이해 | 아이디어 설명과 오늘 만들 작은 화면 | 익명 칭찬 보내기 |
| B2. 칭찬 고르기 | 입력 부담 최소화 | 긍정 문구 4개와 이름 공개 설정 | 한 문구 선택 후 보내기 |
| B3. 칭찬 카드 전달 | 행동 완료 확인 | 선택한 칭찬이 카드 덱에 들어가는 딜 모션 | 닫기, 내 카드 만들기 |
| B4. 내 칭찬 카드 제안 | 받는 사람을 다음 사용자로 전환 | 비어 있는 칭찬 카드 스택 | 내 아이디어 뽑기 |
| B5. 네 장 뽑기 | 새로운 Flow A 시작 | **A1과 화면·동작이 완전히 같음** | 네 장 뽑기 |

### 연결 규칙

B5는 A1의 복사본이 아니라 같은 컴포넌트와 같은 상태 규칙을 사용한다.

## 8. 성장과 수익화

### 성장 루프

네 장 뽑기 → 결과 확정 → 제작 문구 앞부분 확인 → 공유 → 전체 제작 문구 열림 → 친구의 익명 칭찬 → 매일 한 장 뒤집기 → 만든 사람 재방문 → 받은 사람도 네 장 뽑기

### 두 장치의 차이

| 장치 | 시점 | 사용자가 얻는 것 | 제품 역할 |
|---|---|---|---|
| **공유 보상** | 아이디어를 만든 직후 | 전체 제작 문구 | 무료 사용자 획득 |
| **보낸 사람 확인** | 오늘 칭찬을 연 뒤 | 동의한 발신자 이름의 조기 공개 | 수익화 가설 A |
| **다음 카드 미리 보기** | 잠긴 다음 칭찬이 있을 때 | 내일 공개될 칭찬의 조기 열람 | 수익화 가설 B |

잠긴 다음 카드가 없으면 미리 보기 버튼을 노출하지 않는다. 칭찬이 비어 있으면 결제 버튼 없이 공유만 권한다.

### 반복 사용

- 무료 공개 주기는 하루 한 장이다.
- MVP에서는 푸시 알림보다 앱 안의 오늘 카드와 재공유 버튼부터 검증한다.
- 같은 날 여러 장이 오면 도착 순서대로 다음 날짜에 배정한다.

## 9. 성공 기준

### 가장 중요한 지표

**첫 익명 응원 도착률**

> 아이디어를 확정한 사람 중 7일 안에 실제 익명 응원 한 개 이상을 받은 사람의 비율

아이디어가 재미있어도 공유하지 않거나, 공유해도 응원이 오지 않으면 핵심 루프가 작동하지 않은 것이다.

### 선행 지표

1. 첫 방문자의 네 장 완성률
2. 완성한 사람의 아이디어 확정률
3. 확정한 사람의 공유 완료율
4. 공유 링크 방문자의 익명 응원 완료율
5. 응원한 사람의 B5 진입률
6. 응원 상세를 본 사람의 990원 버튼 클릭률
7. 7일 안 재방문율

### 1차 목표값

| 지표 | 1차 통과 기준 |
|---|---|
| 네 장 완성률 | 70% 이상 |
| 결과 이해 성공 | 테스트 참가자 5명 중 4명이 원본과 변화 한 가지를 설명 |
| 공유 완료율 | 확정 사용자 중 40% 이상 |
| 링크 방문 → 응원 완료 | 50% 이상 |
| 응원 → B5 진입 | 20% 이상 |
| 응원 상세 → 990원 클릭 | 10% 이상 |
| 가짜 응원 의심 | 0건 |

### 사용자가 했으면 하는 말

> “새로 생각해내려는 것보다 쉬웠고, 누가 응원했는지 궁금해서 다시 들어왔어요.”

## 10. MVP 범위

### 반드시 만들 기능

1. **검증된 원본 기반 네 장 뽑기**
   - 전체 리롤, 개별 교체, 직접 입력, 색·라벨 연결
2. **쉬운 결과 설명과 제작 문구 공유 보상**
   - 원본을 풀어서 설명, 80~99% 보존 검증, 앞부분 공개·나머지 블러·공유 후 전체 복사
3. **매일 한 장 익명 칭찬 카드**
   - Solitaire 일반 시각 문법, 하루 한 장 무료 공개, 빈 상태 공유, 보낸 사람·다음 카드 가짜 문

### 생성 아이디어의 1인 MVP 하드 게이트

해외 원본의 제품 메커니즘은 80~99% 유지하고 완전히 같아도 허용한다. 단, 결과 아이디어는 다음 조건을 모두 만족해야 카드로 공개한다.

1. 1명이 바이브 코딩으로 반나절~2일 안에 배포할 수 있다.
2. 핵심 경로가 `입력 1개 → 처리 1회 → 결과 1개`다.
3. 화면은 1~3개이며 관리 화면·권한·결제·앱스토어 배포가 없다.
4. 자체 모델 학습, 대규모 크롤링, 여러 외부 서비스 OAuth, 상시 백그라운드 작업이 없다.
5. 관리형 API는 핵심 처리에 필요한 1개를 우선하고 최대 2개를 넘기지 않는다.
6. 성공·빈 상태·실패 상태가 실제 데이터로 작동하며 가짜 사용자 반응을 쓰지 않는다.

범위가 크면 아이디어를 폐기하지 않고 핵심 가치가 처음 발생하는 세로 조각으로 분해한다.

| 큰 원본 범위 | 1인 MVP로 쪼개는 규칙 |
|---|---|
| 여러 플랫폼 자동 수집·발행 | URL·텍스트·CSV 직접 입력 → 미리보기 또는 결과 1개 |
| 상시 모니터링·푸시 알림 | 사용자가 누르는 `지금 확인` → 새 결과 1개 |
| 네이티브 모바일 앱 | 모바일 브라우저에서 동작하는 반응형 웹/PWA |
| 실시간 영상·음성·카메라 처리 | 30초 또는 파일 1개 → 관리형 API 1회 처리 |
| 팀·마켓플레이스·다중 사용자 | 한 사용자·한 기기·한 공유 링크의 단일 경로 |
| 자체 데이터 파이프라인·크롤러 | 사용자가 제공한 실제 입력 또는 수동으로 검증한 소량 데이터 |

분해한 세로 조각도 반나절~2일 안에 실제로 동작시킬 수 없으면 공개 후보에서 제외하고, 핵심 결과를 한 단계 더 줄인다.

### 이번 4주에는 하지 않을 것

- 취향 조사와 취향 기반 가중치
- 카드 고정 기능
- 기존 씨앗·불편·형태·장면·마음 카드
- 공개 피드, 랭킹, 팔로우, 댓글
- 부정 평가와 자유로운 익명 텍스트
- 모닥불·유령·의자·불꽃의 실제 앱 구현
- 가짜 응원·가짜 알림
- 실제 PG 결제
- 발신자 동의 없는 이름 공개
- 이메일·푸시 알림 인프라
- 복잡한 친구 찾기와 연락처 업로드

## 11. 위험한 가정과 안전장치

| 영역 | 가장 위험한 가정 | 먼저 할 검증 | 안전장치 |
|---|---|---|---|
| 가치 | 사용자가 검증된 원본 조합을 새 아이디어보다 더 좋다고 느낀다. | 5명 블라인드 비교 | 원본 충실도와 한 가지 변화를 따로 평가 |
| 사용성 | 아이디어 카드와 칭찬 카드의 역할을 설명 없이 이해한다. | 실제 흐름 테스트 | 생성과 보상 화면을 분리 |
| 성장 | 전체 제작 문구를 위해 실제로 공유한다. | 공유 의사와 실제 공유를 분리 측정 | 공유 취소 시에도 결과는 잃지 않음 |
| 수익 | 보낸 사람 확인 또는 다음 카드 조기 열람 의사가 있다. | 두 가짜 문을 별도 측정 | 실제 칭찬과 잠긴 다음 카드가 있을 때만 노출 |
| 신뢰 | 익명 응원이 진짜 사람에게서 왔다고 믿는다. | 가짜 응원 의심 인터뷰 | 시스템 생성 응원 금지, 전송 시각과 상태 명확화 |
| 개인정보 | 이름 공개가 동의 범위 안에서 작동한다. | 공개 동의 문구 이해 테스트 | 영구 익명은 결제해도 공개 금지 |
| 콘텐츠 | 원본 카드가 구체적이고 다양하다. | 감사 통과율·플랫폼 분포 확인 | TrustMRR·App Store·상세 조사 근거 필수 |
| 기술 | 로그인 없는 응원 제출이 악용되지 않는다. | 중복·자동 제출 테스트 | 속도 제한, 중복 방지, 신고·차단 준비 |

## 12. 1주차 검증 실험

코드를 더 만들기 전에 정적 HTML로 Flow A 5화면과 Flow B 5화면을 한 줄씩 보여준다.

참가자 5명에게 설명 없이 다음을 시킨다.

1. 네 카드가 각각 무엇인지 말한다.
2. 결과에서 원본과 달라진 한 가지를 찾는다.
3. 잠긴 제작 문구를 여는 방법을 찾는다.
4. 오늘 무료 카드와 잠긴 다음 카드의 차이를 말한다.
5. 보낸 사람 확인과 다음 카드 미리 보기의 차이를 설명한다.
6. 받은 사람이 어디서 자신의 네 장 뽑기를 시작하는지 찾는다.

실패 기준:

- 2명 이상이 취향 설정을 먼저 찾는다.
- 2명 이상이 잠긴 카드에 칭찬 문구가 이미 노출된다고 생각한다.
- 2명 이상이 칭찬이 없는데도 결제해야 한다고 생각한다.
- 공유 보상과 990원 결제를 같은 상품으로 이해한다.
- 아이디어 조합 카드와 익명 칭찬 카드를 같은 기능이라고 이해한다.

## 최종 결정 요약

1. 핵심 제품은 **검증된 제품 기반 네 장 뽑기 + 익명 응원**이다.
2. 취향 조사와 고정은 폐기한다.
3. 카드 분류는 **검증된 원본 / 돈 낼 사람 / 필요한 순간 / 한 끗 변화**다.
4. 원본은 제품 이름만 쓰지 않고 고등학생도 이해하게 풀어 쓴다.
5. 실제 칭찬을 **매일 한 장 무료 카드**로 공개한다.
6. 칭찬이 비어 있으면 결제가 아니라 공유를 권한다.
7. 보낸 사람 확인과 다음 카드 미리 보기는 서로 다른 결제 가짜 문이다.
8. Flow A와 Flow B는 각각 5화면이며 B5는 A1과 동일하다.
9. 모바일 퍼스트로 만들되 데스크톱 비교 화면도 지원한다.
10. 모닥불 구현은 제거하고 문서·정적 HTML 레퍼런스로만 보존한다.
