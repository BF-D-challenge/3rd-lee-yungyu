## 1. v2에서 바뀌는 것 (요약)

| 항목 | v1 | v2 |
|------|----|----|
| 슬롯의 출발점 | 순수 랜덤 조합 | **씨앗 입력**: 내가 좋아하는 것/잘 아는 것 1개를 먼저 받고, 그 축은 고정 |
| 슬롯 릴 수 | 5축(누구·문제·상황·형태·심리) 전부 노출 | **표면 3릴**: 씨앗(고정) × 누구의 불편 × 형태. 상황·심리는 백엔드 조합 규칙으로 숨김 |
| 제품의 약속 | "아이디어 1개를 뽑아준다" | **"내 것에서 나온 아이디어 + 수요 신호까지"** — 투표 카드가 수요 확인 도구로 승격 |
| 계정 | 비공개 URL (로그인 없음) | **Supabase Auth 기본 로그인** (구글 OAuth + 이메일 매직링크) |
| 유료 상품 | 실행 플랜 990 / 오늘 패스 1,900 / 반응 상세 | 유지 + **수요 리포트**가 반응 상세를 흡수 |

**리서치 판정 (2026-07-06, [통합 요약](../../.claude/knowledge/research/oneul-haebolkka-demand/SUMMARY.md))**: 수요 신호 **중~강** — "뭘 만들지 모르겠다" pain은 3개 언어권에서 실증. 단 "AI 아이디어 생성기" 형태에는 냉소가 강하고(유사 툴 도배, "툴 말고 유저 5명과 대화하라"), 순수 생성의 지불 의사는 0에 수렴(IdeasAI 실패). 성립하는 과금은 1회성 소액 리포트(DimeADozen $9)로 본 설계의 990·1,900원과 정합. → **포지셔닝 규칙: "아이디어 생성기"라고 말하지 않는다. "내 것에서 시작해 오늘 바로 만들게 하고, 12명에게 물어봐주는 도구"로 말한다.** 커뮤니티가 처방하는 대안("유저와 대화하라")이 곧 투표 카드 기능이므로, 냉소 진영의 언어를 그대로 카피로 쓴다.

**왜 바꾸나 — 페르소나 평가(§2)의 결론**: 순수 랜덤 슬롯은 4개 페르소나 중 1명만 만족시킨다. 1인 빌더는 "아무거나"가 아니라 **자기가 좋아하고 잘 아는 것**을 만들고 싶어 하고, 만들기 전에 **수요가 있는지** 알고 싶어 한다. 씨앗 고정과 수요 리포트가 이 둘을 정면으로 받는다. 또한 결제 관점에서, 씨앗이 들어가야 잠긴 실행 플랜이 "ChatGPT 무료 대체재"가 아니라 "내 것에 대한 맞춤 플랜"이 된다(결제 성립의 전제).

## 2. 고객 페르소나 4명 평가

공통 전제: AI 코딩 툴은 열 수 있음. 평가 대상은 두 설계 — **A: 순수 랜덤 슬롯(v1)** vs **B: 씨앗 기반(v2)**.

### P1. 지민 — 29세, 주니어 마케터, Lv.1 "진짜 막막형"
- 상태: 사이드프로젝트 하고 싶다는 말만 6개월째. 좋아하는 건 있음(러닝, 카페 투어)인데 그게 아이디어가 된다는 생각을 못 함.
- A 평가: 슬롯 재미로 3~4회 돌리다가 "이게 내 얘기랑 뭔 상관이지" → 이탈. 결제 근거 약함.
- B 평가: "러닝"을 씨앗으로 넣는 순간 결과가 전부 자기 얘기가 됨. **4명 중 씨앗 입력이 가장 쉬운 사람** — 좋아하는 건 누구나 말할 수 있다.
- 결제 포인트: 실행 플랜 990원 (확신 구매). 카드 공유 대상: 러닝 크루 → 투표 모수 있음.

### P2. 현우 — 33세, 백엔드 개발자 5년차 "아이디어 과잉·확신 부족형"
- 상태: 노션에 아이디어 30개. 못 고르고, 골라도 "이거 수요 있나?"에서 멈춤. 아이디어 생성기를 장난감 취급할 위험이 가장 큰 사람.
- A 평가: "랜덤이 내 노트보다 나을 리 없다" → 냉소, 이탈.
- B 평가: 씨앗에 자기 도메인을 넣고 **투표 카드 = 수요 조사 도구**라는 걸 아는 순간 태도가 바뀜. 이 사람에게 제품의 본체는 슬롯이 아니라 **수요 리포트**.
- 결제 포인트: 반응 상세/수요 리포트 (누가·왜 써볼래를 눌렀는지). 지불 의사 높음.

### P3. 소연 — 26세, 프리랜서 디자이너 "취향 기반·감성형"
- 상태: 빵집 투어 계정 운영. "내가 좋아하는 걸로 뭔가 만들고 싶다"가 동기의 전부. 랜덤이 주는 "남의 아이디어" 느낌에 거부감.
- A 평가: 첫 결과가 물류 SaaS면 즉시 이탈.
- B 평가: 씨앗 고정이 필수 조건. 추가 요구: **카드가 인스타 스토리에 올릴 만큼 예뻐야** 함 — 이 사람이 바이럴 층의 핵심 발신자(트리꾸미기 스토리 문법의 담당자).
- 결제 포인트: 오늘 패스(탐색을 오래 함). 공유 전환율 4명 중 최고.

### P4. 태호 — 41세, 물류 15년 직장인 "도메인 전문가·시간 빈곤형"
- 상태: 업계 비효율을 매일 봄. 잘 아는 건 확실한데 시간이 없고, "만들었는데 아무도 안 쓰면"이 최대 공포.
- A 평가: 슬롯 자체를 유치하다고 느낄 위험. 랜덤은 논외.
- B 평가: 씨앗("물류 검수")을 넣고 나온 카드를 **업계 동료 단톡방에 뿌려 수요 투표**를 받을 수 있다는 점이 유일하고 강한 후크.
- 결제 포인트: 가격 민감도 최저 — 시간을 아껴주면 990원이 아니라 9,900원도 냄. **가격 상방 실험 대상** (어둠의 MBTI 9,900원 문법, 53:18).

### 평가 종합

| | P1 지민 | P2 현우 | P3 소연 | P4 태호 |
|---|---|---|---|---|
| A: 랜덤 슬롯 | △ 재미로 시작, 결제 약함 | ✗ 냉소 | ✗ 거부감 | ✗ 논외 |
| B: 씨앗 기반 | ◎ | ○ 수요 리포트가 후크 | ◎ 씨앗+예쁜 카드 | ○ 단톡방 수요조사 |
| 주 결제 상품 | 실행 플랜 990 | 수요 리포트 | 오늘 패스 1,900 | 수요 리포트(상위가) |
| 바이럴 기여 | 러닝 크루 | 낮음 | **최고(스토리)** | 단톡방(B2B 밀도) |

→ **랜덤 슬롯은 P1 전용 설계였다. 씨앗 기반으로 바꾸면 4명 전원에게 진입로가 생기고, 결제 상품도 페르소나별로 갈라진다.**

## 3. 사용자 전체 여정 v2

### 3.1 카드 단순화 — "6장 뽑기"의 인지 부담 제거

기존 5축을 사용자에게 다 보여주는 건 P1에게 과부하, P4에게 유치함이었다. v2:

- **표면 3릴**: `[씨앗 — 내 것, 고정]` × `[누구의 어떤 불편]` × `[형태(앱/봇/링크도구…)]`
- `상황`·`심리` 축은 삭제가 아니라 **백엔드 조합 규칙으로 이동** — 결과 문장의 다양성은 유지하되 사용자가 릴로 조작하지 않는다.
- 리믹스도 단순해짐: 고정할지 말지는 2릴에 대해서만 결정.

### 3.2 여정 다이어그램

```
[온보딩 — 1회, 전부 객관식(탭 4번)]
  S0-a "어느 쪽이에요?"  [ 좋아하는 걸로 ] / [ 잘 아는 걸로 ]
  S0-b 대분류 칩 12개 (운동/음식/여행/콘텐츠/게임/재테크/…
        또는 직군: 마케팅/물류/교육/의료/…)
  S0-c 소분류 칩 6~8개 (운동 → 러닝/헬스/클라이밍/골프/…)

     (비로그인으로 진행 가능 — 로그인은 나중, 3.3 참조)

        │                                              나도 뽑아보기" → S0
        ▼
  S6 수요 대시보드 (마이페이지, 로그인)
     무료: 투표 수 카운트 "12명 중 7명이 써볼래"
     유료: 수요 리포트 — 누가·왜(한마디), 반응별 분해,
           "만들어라/보류" 판정 요약                        [결제 ③]
```

여정의 서사가 바뀐다: v1은 "아이디어를 뽑는다"에서 끝났지만, v2는 **"내 것에서 뽑고 → 수요를 확인하고 → 플랜을 산다"** — 페르소나 4명 전원의 동기(내 것 + 수요 불안)를 관통한다. 바이럴·리텐션·결제 배선(v1)은 전부 유지되고, 공유의 명분이 "자랑"에서 **"수요 조사"로 바뀌어 더 이타적**이 된다(48:57): 받는 사람은 도와주는 역할을 얻는다.

### 3.3 로그인 설계 (Supabase Auth)

- **방식**: 구글 OAuth + 이메일 매직링크. (카카오 로그인은 다음 버전 유지 — PG·심사 등 마찰 대비 효익 낮음)
- **로그인 요구 시점 = 카드 발행(S3)**. 원칙: 바이럴 퍼널(수신자 투표)에는 절대 로그인을 두지 않는다 — 수신자 마찰은 K를 직접 깎는다. 슬롯 체험도 비로그인 허용(세션 저장), 발행하려는 순간에만 계정을 만들게 한다. 이 시점의 사용자는 이미 확정까지 온 고관여 상태라 이탈이 가장 적다.
- **로그인이 주는 것**: 내 카드 대시보드(S6), 투표 알림 이메일, 데일리 스핀 카운트의 기기 간 유지, 결제 이력.

### 3.4 Supabase 스키마 초안

```
profiles   (id=auth.uid, seed_text, weekly_hours, skill_level, created_at)
ideas      (id, owner_id→profiles, seed_text, combo_json, title, oneliner,
            plan_unlocked bool, share_slug unique, created_at)
votes      (id, idea_id→ideas, vote_type enum[try|empathy|meh],
            comment text null, voter_fp text, created_at)
            -- 비로그인 투표: voter_fp = 브라우저 지문+IP 해시, (idea_id, voter_fp) unique
spins      (user_or_session_id, date, count)   -- 데일리 5회 캡
purchases  (id, user_id, idea_id null, product enum[plan|day_pass|demand_report],
            amount, status, created_at)
```

- RLS: `ideas` 읽기는 share_slug 경유 공개 / 쓰기는 owner. `votes` insert는 익명 허용 + rate limit(Edge Function), 읽기 상세는 owner + 결제 확인. `purchases`는 owner만.
- 결제 연동(PG)은 강의에서 김시현 영역이라 본 대본에 근거 없음 — 구현 시 별도 검토.

## 6. 4주 일정 v3 (2026-07-06 시작) — 순서를 뒤집는다: 실측이 무엇을 진짜로 만들지 결정

**v3 재배열 원칙(적대적 진단 반영)**: v2는 가장 어렵고 에러 다발인 작업(Auth·RLS·PG)을 2~3주차에 back-load해서, 유일하게 학습을 만드는 4주차 실측이 슬립 완충재로 소모되는 구조였다(32:57 두 번째 벽의 교과서적 실패). v3은 **1주차 말~2주차 초에 "전부 가짜인 end-to-end 루프"를 실유저에게 투하해 실측부터** 한다. 그 데이터가 "어느 한 단계"가 진짜 병목·진짜 가치임을 증명한 뒤에야, 그 한 단계에만 진짜 Supabase/PG를 3~4주차에 판다. 80h는 gross이고 유효 60~65h이므로 예산도 이 순서를 강제한다.

| 주차 | 목표 | 산출물 |
|------|------|--------|
| 1주차 7/6–7/12 | 정적 MVP + end-to-end 가짜 루프 | 씨앗 입력(좁은 택소노미 3~4 대분류 + [직접 쓸래요]) + 씨앗-conditioned 3릴 슬롯 + 확정 + **가짜 문 3개 전부(플랜 990·오늘패스·수요 리포트)** — 클릭·이메일만 측정. 카드 공유=정적 이미지/URL을 유저가 직접 카톡 발송, 투표=구글폼(무료 대체재를 내가 손으로 흉내), 수요 리포트=폼 응답 보고 **수기 작성해 전달(Wizard-of-Oz)**. 콘텐츠(골든 조합 ~20개, 씨앗 태그별 allowlist)를 1주차에 저작. **Day 1부터 페르소나별 5명 병렬 모집**(리크루팅이 진짜 long-pole) |

## 7. 리스크 v3 (적대적 진단 종합 — 중복 병합)

v3에서 적대적 진단이 발견한 high 심각도 리스크를 결제 배치·바이럴·조합엔진·범위의 4개 축으로 병합해 재정리한다. **살릴 것: 씨앗 고정 + 수요 리포트를 핵심 단일 결제로. 미룰 것: Supabase Auth 전체·결제 2·3지점 동시 검증·리텐션 인프라.**

### 7.A 결제 배치 — "잘 팔릴 상품은 느린 퍼널 뒤, 안 팔릴 상품은 빠른 순간에" 역배치 (high)
- **R1. 실행 플랜(②·990)이 확정 정점을 지나 배치됨.** v2 §3.2는 확정→발행→투표대기→S6→그다음 S4 순이라, 아이디어에 가장 뜨거운 순간(확정 직후)에 결제를 안 받고 며칠짜리 비동기 대기가 식힌 뒤 판다(강의 52:41 대운 즉시 잠금과 반대). **교정: 확정(S1) 직후 두 갈래 분기 — [바로 만들 플랜 990] / [먼저 수요 물어보기]. 실행 결제를 비동기 대기의 인질로 두지 않는다.**
- **R2. 지불의사가 실증된 유일 상품(수요 리포트, DimeADozen $9형)이 가장 느린 퍼널 뒤에 있음.** 리포트는 투표 5+ 도달에 전적 의존(순환 의존). **교정: 핵심 결제를 수요 리포트 하나로 정하되 "카드 발행 시점 선결제(투표 모이면 리포트 받는 예약형)"로 비동기 지연을 결제 앞으로 당김(어둠의 MBTI 53:18 선결제 문법). "즉시 수요진단"(집합 데이터·휴리스틱, 1,900)과 "실제 리포트"(투표 축적 후, 고가)를 분리.**
- **R3. 오늘패스가 "무제한 스핀"을 팖 = ChatGPT가 무한 무료로 하는 행위.** 순수 생성 지불의사 0(IdeasAI). 게다가 스핀 캡의 유일한 강제력(데일리 5회)이 신원 벽 없어 시크릿창으로 0원 우회됨. **교정: "무제한 스핀" 문구 제거 → "오늘 확정→플랜→발행까지 끝내는 실행 패스"로 재포지셔닝(결과물 묶고 스핀은 부산물). 캡을 결제 트리거로 쓰지 말고 "이미 돌린 5개 결과의 보존·비교"에 과금(우회=산출물 손실).**
- **R4. 990 플랜 산출물(이름·한줄·MVP·4주플랜·프롬프트)을 ChatGPT가 무료로 뱉고, S1 확정에서 이름·한줄을 이미 무료 공개함.** '맞춤이라 대체재 아님' 논리가 프롬프트 한 줄로 무너짐. **교정: 무료 노출을 '분위기/카테고리'까지로 낮추고, 990을 ChatGPT가 못 만드는 것(같은 씨앗 태그 유사 카드들의 투표 전환율·생존 벤치마크, 사전검증 금지조합 회피 근거 = 집합 데이터 기반 플랜)으로 교체. 백엔드 상황·심리 축+시간/레벨을 플랜 생성 프롬프트에 전량 주입해 '내 것 맞춤'의 실질 근거 확보.**
- **R5. 수요 리포트가 무료 카운트("12명 중 7명")에 양방향 자기잠식.** 좋으면 무료로 만족, 나쁘면 안 삼. **교정: 무료는 "N명 도착(내용 비공개, 첫 1표 미리보기)"까지만. 카운트 자체를 유료 자물쇠에 포함(51:56 핵심가치 무료체험 금지). 표본 부족 카드(5표 미도달)엔 리포트 판매 노출 안 함 → 5표 도달 시 결제 트리거. 표본 3개여도 씨앗 태그 집합 벤치마크로 맥락 있는 판정.**
- **R6. 990↔1,900 앵커 충돌 / 스핀 페이월의 무료 우회("공유 +3").** 플랜 990이 오늘패스 1,900의 스핀 가치를 910원으로 노출; 공유 +3은 셀프-게이밍으로 무한 무료화. **교정: 두 결제지점을 같은 세션 동시 노출 금지. 공유 +3은 "발송"이 아니라 "타인 지문 투표 1건당 +1"로 조건화((idea_id,voter_fp) unique+지문 이질성) → 우회가 곧 K 상승. ③ 가격은 1,900=요약판정 / 4,900=투표자별 코멘트 원문 / 9,900=업계 벤치마크로 가치사다리 명시.**
- **R7. 투표 무결성 = 리포트 주 구매자(P2 회의주의자)의 최대 불신 지점.** 지문+IP 셀프 부양 가능 → "내가 조작할 수 있는 데이터"에 돈 안 냄. **교정: 리포트 표지에 무결성 신호("검증된 고유 투표자 N명, 자가·중복 제외") 노출, 판정이 항상 '만들어라'로 안 기울게 '보류/재설계' 케이스 UI 강조, 냉소 진영 언어 그대로 카피화.**

### 7.B 바이럴 루프 — 공유물이 "남의 아이디어"라 K가 구조적으로 낮음 (high)
- **R8. '내 아이디어에 투표해줘'는 여전히 이기적 공유(품평 노동을 give).** 990 사주·MBTI는 수신자도 '내 것이 궁금'해 자기 것을 함(157줄 애니팡, 카톡은 채널일 뿐). **교정: 투표 화면에 "너라면 이 씨앗으로 뭘 만들까?" 미니 슬롯을 투표와 동시 노출(발신자 루프 진입을 투표 그 자체에 합침). 투표 후 수신자에게도 즉각 보상(씨앗 조합 다른 반응 미리보기/즉석 취향 진단). 소연(P3)이 요구한 '인스타 스토리용 예쁜 카드'를 결제·일정의 명시적 산출물로 승격 — 공유물 자체가 누르고 싶어야 함(48:57).**
- **R9. 수신자→발신자 전환에 두 마찰: 온보딩 4탭 재통과 + S3 로그인 벽.** 트리꾸미기 '내 트리 만들기'는 원클릭(50:44 "여기 끊기면 사이클이 안 커진다"). **교정: 전환 진입 기본값을 방금 투표한 카드의 씨앗·문맥 프리필 1탭 슬롯으로, 온보딩은 발행 순간으로 지연, 전환 유저엔 구글 원탭만.**
- **R10. 첫 발신자 확보가 4주차 코호트 1~2곳뿐이라 얇음.** 커뮤니티에서 '아이디어 생성기' 피칭 글은 점수 0~2로 죽음(리서치). **교정: 첫 1,000명은 바이럴에 기대지 말고 1주차 수동 실험으로 문제공감형 콘텐츠 채널 확보. K는 '사주형 후크'가 검증된 다음에만 기대라.**
- **R11. 스핀캡 페이월의 '공유 +3'가 확정도 못 한 부끄러운 카드를 공유시킴(P1 막막형 빈발).** **교정: '자랑'이 아니라 '친구야 이 중 뭐가 나아?' 투표 요청 프레임(이타적)으로.**

### 7.C 조합엔진 — 씨앗-conditioning 없는 랜덤은 기본값이 붕괴 (high, 다음 단계 엔진 최우선)
- **R12. 불편·형태 릴이 씨앗과 독립 랜덤이면 조합공간 대다수가 무의미(러닝×물류검수 불편×슬랙봇).** 씨앗 고정의 존재 이유("결과가 전부 내 얘기")가 무너짐. **교정: 표면 3릴 중 씨앗만 고정이 아니라 불편·형태 모두 씨앗-conditioned 풀에서만 추출. allowlist(씨앗 태그별 허용 불편·형태 화이트리스트)를 blocklist보다 우선하는 1급 자료구조로, 각 태그당 사전검수 골든 조합 N개 확보를 2주차 필수 산출물(nice-to-have 아님)로 격상 → 1주차는 ~20개로 시작.**
- **R13. 상황·심리를 백엔드로 숨긴 결정이 '뻔함'(Strava 재발명)과 얕은 플랜을 낳음.** §5 최대 가정('만들어볼 만하다')이 여기서 깨짐. **교정: 결과 문장 템플릿에 상황·심리 슬롯을 필수 주입(씨앗×불편 조건부 규칙). 리롤 시 숨은 상황/심리 축도 흔들어 '뻔함 탈출구' 확보 + 세션 내 중복 회피 큐로 체감 조합 수 배가.**
- **R14. 리롤이 순수 랜덤이면 인접 개선이 아니라 무관 점프 → 캡 소진 시점엔 '랜덤 쓰레기' 판정 끝나 결제 거부.** **교정: 리롤 기본을 '고정 축 유지 + 나머지 1축만 인접 이동'(원본 PRD 리믹스)으로. 페이월 앞 첫 2~3스핀은 골든 조합 먼저 노출해 신뢰 확보 후 랜덤 개방.**
- **R15. 형태 릴이 컨테이너(앱/봇) 수준이라 정보량 0 + 씨앗×형태 궁합 붕괴.** **교정: 형태를 '기능 패턴'(기록·추적/매칭/챌린지·리더보드/자동 리마인드/큐레이션)으로 재정의하고 씨앗×불편 종속 변수로. S0-a 좋아하는것/잘아는것 분기를 형태 풀(감성→앱·스토리카드 / B2B→봇·대시보드) 결정에 활용.**
- **R16. [직접 쓸래요] 자유입력 씨앗은 태그 밖이라 conditioning 규칙 적용 불가 → 궁합 붕괴의 뒷문.** **교정: 자유입력 씨앗은 발행 전 LLM/임베딩으로 최근접 태그 매핑 후 화이트리스트 상속. 신뢰도 임계 미만이면 조합엔진 대신 LLM 단독 생성 경로로 분기. 조합엔진에 그대로 흘려보내기 금지.**

### 7.D 온보딩·씨앗 택소노미 — 객관식 칩이 고가치 페르소나를 탈출구로 강등 (high/med)
- **R17. P1 카페투어·P3 빵집투어는 12칩의 교집합·틈새라 뭉개지고, P2(IT/개발)·P4(물류검수 니치)는 첫 화면부터 [직접 쓸래요]로 내몰림(수요 리포트 주결제층인데).** **교정: 복수 태그 조합(음식+여행+콘텐츠) 허용 + 인기 씨앗 프리셋 칩('카페투어','빵집투어','러닝크루')을 소분류 상단 노출. 직군 칩에 IT/개발 추가, '잘 아는 걸로' 경로는 자유입력을 탈출구가 아닌 1급 입력으로 승격(자동완성·최근 씨앗). 소분류 칩 추상도 균일 정규화('러닝' 수준), 초광범 칩(재테크·콘텐츠)은 소소분류 강제.**
- **R18. P2 pain은 사후형('노션 30개 중 이거 수요 있나')인데 반드시 슬롯을 통과해야 카드 발행 가능.** **교정: '이미 만들 것 정했어요 → 바로 수요 카드 만들기' 진입점으로 슬롯 우회.**
- **R19. 씨앗 온보딩 콘텐츠(12×8×2 ≈ 80~100칩)의 저작 부담 과소평가.** **교정: 1주차는 좁은 택소노미(대분류 3~4)만 + 탈출구로 출시, 전체 그리드는 '칩이 자유입력보다 낫다' 검증 후 확장.**
- **R20. P4(도메인 15년차)에게 슬롯 UI 자체가 아마추어스러움(§2 자인).** **교정: '잘 아는 걸로' 분기부터 톤을 놀이→도구로 전환, 슬롯 은유 약화, 결과 근거 앞세움.**

### 7.E 로그인·인프라·범위 — critical path와 최대 병목이 겹침 (high)
- **R21. 수요 루프 백엔드 전체(Auth+스키마+RLS+발행+공개카드+투표3종+대시보드)가 2주차 20h 한 칸에 압축 = critical path와 최대 병목 중첩.** **교정: 2주차 진짜 백엔드를 걷어내고 1~2주차는 Wizard-of-Oz(카톡 직접 발송·구글폼 투표·수기 리포트). Supabase 실구현은 '공유가 실제로 일어난다' 증명 후 3주차만.**
- **R22. 매직링크(2주차)+투표 알림 이메일(3주차)의 숨은 이중 SMTP 의존성(DNS·SPF/DKIM, 며칠 블로커)을 0원 계상.** **교정: 매직링크 MVP 삭제 → 구글 원탭 단독(70% 조건 기다리지 말고 처음부터). 투표 알림 이메일 폐기 → 재방문 URL 본 플랜. 이메일 인프라 4주 안에 미접촉.**
- **R23. 3주차 결제 연동(PG)은 두 번째 벽(32:57)이자 강의 밖 영역인데 3주차 1/4만 배정.** 한국 PG는 사업자등록·웹훅 멱등성·15~25h 에러 다발. **교정: 세 결제 지점 전부 4주차까지 가짜 문. 진짜는 Stripe Payment Link/계좌이체+수기 언락. 검증 대상은 결제 '기술'이 아니라 지불 의사(클릭률·이메일). PG 진짜 연동은 챌린지 후 별도 스프린트.**
- **R24. RLS+익명투표+rate limit이 한 줄이지만 실제 5개 별개 난제(공개 읽기·소유자 게이팅·익명 insert·지문 unique·Edge 레이트리밋), solo 빌더 사일런트 실패 지점.** **교정: MVP는 votes 단일 공개 테이블 permissive insert + 클라이언트 중복 방지(localStorage)만. 게이팅·레이트리밋은 어뷰징 관측 후. finding대로 투표를 구글폼으로 빼면 2주차에서 전부 소거.**
- **R25. 리텐션 인프라(데일리 캡 리셋·투표 알림·기기간 유지)가 1회성 결제 철학과 배치.** 1인 빌더 리텐션은 기업도 어려움(57:13, 58:40). **교정: 리텐션 기계를 4주 범위에서 걷어냄. 재방문은 '투표 모였으니 결과 확인' 1회성 트리거로 족함.**
- **R26. 카드 데이터 50+ 콘텐츠 저작(8~12h)이 Auth/RLS와 같은 2주차 칸에서 시간 다툼.** **교정: 콘텐츠 저작을 1주차로 이동(정적이라 조기 가능), 50이 아니라 ~20으로 시작 후 실측이 확정률 만드는 태그 알려주면 확장.**
