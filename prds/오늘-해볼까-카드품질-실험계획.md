# 카드 조합 콘텐츠 품질 — 가설 실험 계획 (2026-07-07)

> 실행 설계서(하위 에이전트용 태스크 스펙 T1~T7): [`docs/experiments/card-quality/DESIGN.md`](../docs/experiments/card-quality/DESIGN.md)

> 문제: 슬롯 조합 결과 카드의 내용이 "너무 별로". 여러 방향을 가설로 세워 짧고 가볍게 생성 → 사용자가 블라인드 채점으로 검증 → 승자 방향만 전량 롤아웃.
> 사용자 확정 사항: 품질 기준 = **"진짜 레퍼런스·매출이 있는 것처럼 그럴듯하게, 90% 비슷 + 10% 다르게"** · LLM은 **오프라인 배치만** · 검증은 **블라인드 비교 시트 20조합 × 4안** · 장면·마음 축은 **자연스러울 때만 반영** · **타로/오늘의 운세 컨셉**도 후보에 포함.

---

## 1. 진단 (코드·데이터 실측)

| 원인 | 근거 |
|---|---|
| ① 뽑기 가능한 조합 1,649개 중 사전 검수(골든) 카드 88장 = **5.3%**. 나머지 95%는 매드립 템플릿 문장 | `data/combos.json` allow × golden 계산. 템플릿: `{situation}에 {psych}을 느끼는 {target}을 위해 — {seed}의 "{painShort}"를 {format}(으)로 푸는 도구` (`lib/draw.ts:90`) |
| ② 장면·마음 축이 씨앗·불편과 무관하게 **랜덤 접착** → "출퇴근 짬시간에 골프 스윙 봐달라" 같은 부정합 | `drawSituation`/`drawPsych`는 전체 풀에서 무조건 랜덤 (`lib/pools.ts:133`) |
| ③ 카드에 **"실존하는 수요"라는 근거가 없음** → 그럴듯함 부족. 골든 88장의 품질 평가에서도 #17(진부)·#18(니치)·#19(ChatGPT 일반론)이 3라운드 내내 최하위 | `docs/idea-quality-eval-report.md` |
| ④ 비골든 조합은 title/MVP가 null → 문장 한 줄만 나옴 (정보량 부족) | `lib/draw.ts:79-86` |

활용 가능 자산: `research/trustmrr-acquire/ideas.jsonl` — **실제 30일 매출 수치가 붙은 제품 1,863개** (카테고리·problem·revenue_30d_value 포함). "진짜 레퍼런스" 원료로 사용.

## 2. 가설 4방향 (블라인드 시트의 4개 열)

| 열 | 방향 | 내용 | 검증하려는 질문 |
|---|---|---|---|
| **H0** | 현행 (대조군) | 지금 코드가 내는 그대로 (골든이면 골든, 아니면 템플릿 문장) | 베이스라인 |
| **H1** | 근거 명시형 | 조합마다 trustmrr 코퍼스에서 유사 실제품을 앵커로 매칭 → "검증된 메커니즘 90% + 이 씨앗·니치로 10% 비틀기"로 생성. 카드 하단에 **근거 한 줄 명시** ("비슷한 유형이 월 $4k 매출 — 그것의 동네 버전") | LLM+실제 근거가 템플릿을 이기나? 숫자 노출이 신뢰를 만드나? |
| **H2** | 근거 암시형 | H1과 동일 생성이되 **제품명·매출 숫자는 숨김** — "해외에선 이미 돈 내고 쓰는 유형" 수준의 문체만 | 숫자 명시(H1)가 더 그럴듯한가, 오히려 과장광고처럼 보이나? |
| **H3** | 타로 운세형 | **"오늘의 운세" 내레이션 컨셉** — 타로 리딩체로 카드가 사용자에게 말을 걸고("오늘 당신에게 온 카드는…"), 아이디어·타깃·오늘 할 일을 운세처럼 풀어줌. 근거 한 줄은 H1 방식으로 결합 | 운세 컨셉이 재미·의외성을 더하나, 정보를 가리나? (기존 타로 카드아트 UI와 시너지) |

공통 생성 규칙 (H1~H3): 장면·마음은 씨앗×불편과 정합할 때만 문장에 녹이고 어색하면 생략 · `format.short` 대신 `format.action` 사용 · title + oneliner + target + MVP 4종 풀세트 생성 · 품질 기준 4개(니치 구체성 / 오늘 만들 실행감 / 의외성·재미 / 문장 자연스러움)를 생성 프롬프트에 루브릭으로 주입.

## 3. 실험 설계

**표본 — 20조합, 고정 시드 층화 추출:**
- like 트랙 10 · know 트랙 10, 카테고리 중복 최소화
- 골든 히트 조합 4개 포함 (현행 최선과도 비교되도록)
- 장면·마음 축이 포함된 조합 10개 (정합성 처리 검증용, "출퇴근×골프" 같은 극단 부정합 2개 의도적 포함)

**생성 — 오프라인 배치, 이번 세션 안에서:**
- 앵커 매칭: ideas.jsonl 1,863개를 카테고리·키워드로 사전 필터 → 조합당 후보 3~5개를 생성 에이전트에 전달, 에이전트가 최적 앵커 선택 후 90/10 작성
- 20조합 × 3방향(H1·H2·H3) = 60건 병렬 생성. API 키·런타임 비용 0 (세션 에이전트로 배치)
- H0는 현행 코드 로직 그대로 재현해 조립

**블라인드 프로토콜:**
- 행마다 4개 열 순서를 셔플, 방향 라벨 제거
- 산출물 ① `docs/experiments/card-quality-blind-sheet.md` (채점용) ② 정답 키는 별도 파일로 분리 (채점 전 열람 금지)
- 채점 방식: 행별 **승자 1개 선택** + 선택적으로 "이건 탈락" 표시. 예상 소요 15~20분

## 4. 판정 규칙

- 한 방향이 **12행 이상(60%) 승리** → 그 방향 확정 채택
- H1 vs H2 우열 → 카드 스키마에 앵커 필드(제품·매출) 노출 여부 결정
- H3 승리 → 카드 카피를 넘어 결과 화면 컨셉 자체를 운세 리딩으로 전환하는 후속 검토
- 과반 승자 없음 → 사용자 탈락 사유 코멘트를 모아 2차 가설 (예: H1+H3 하이브리드) 후 10행 재채점

## 5. 채택 후 롤아웃 (검증 통과 후 별도 착수 — 이번 범위 아님)

1. 승자 포맷으로 allowlist 1,649조합 **전량 배치 생성** → 모든 뽑기가 사실상 골든이 되도록 combos.json(또는 분리 파일) 확장
2. 기존 29항목 루브릭 자동 평가를 품질 게이트로 재사용 (통과 못 한 카드만 재생성)
3. `sentence()`/`confirm-branch`/결과 카드 UI 반영 (H1 승리 시 근거 필드 추가, H3 승리 시 운세 내레이션 UI)
4. 장면·마음 정합성 규칙을 뽑기 단계(`pools.ts`)에도 반영할지 재검토

## 6. 리스크

- 영문 코퍼스 앵커 → 한국 니치로 옮길 때 번역투·문화 부정합 가능 (생성 프롬프트에서 "한국 생활어로 재해석" 강제)
- H3 운세체가 재미는 있어도 4대 기준 중 '실행감'을 흐릴 수 있음 — 그래서 H1과 동일한 근거·MVP 정보를 유지한 채 톤만 바꿈
- 20행 표본에서 우열이 애매할 수 있음 → 판정 규칙 4에 재채점 경로 마련
